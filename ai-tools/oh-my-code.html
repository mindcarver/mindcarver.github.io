核心概念

什么是 Oh My OpenCode？

Oh My OpenCode 是 OpenCode 的插件，通过多 Agent 协作系统，将单一的 AI 编程助手升级为具备高度自主性的 AI 开发军团。

关键词：ultrawork / ulw

在提示词中加入 ultrawork 或 ulw，即可触发全自动工作模式：

能力说明
自动探索分析代码库结构和依赖关系
自动计划拆解任务，制定执行步骤
自动分配调用专家 Agent 处理各自专长
自动验证运行测试和诊断确保质量
持续迭代任务未完成则自动优化直到完成

两种工作模式

Ultrawork 模式

适合快速任务、明确目标、信任 AI 自主决策的场景。

在提示词中加入 ulw 或 ultrawork：

工作流程：

用户输入 ulw 任务
    ↓
Sisyphus 接收，分析意图
    ↓
自动探索代码库模式
    ↓
并行调用专家 Agent (Hephaestus, Oracle, Librarian...)
    ↓
验证结果（测试、诊断）
    ↓
未完成？继续迭代
    ↓
100% 完成 → 结束

Prometheus 模式

适合复杂项目、关键生产变更、需要文档化决策的场景。

执行步骤：

  1. 进入 Prometheus 模式

    在 OpenCode TUI 中按 Tab 键

  2. 描述工作

    我要为电商系统添加完整的支付流程，支持多种支付方式

  3. 接受采访

    Prometheus 会通过问题来明确需求：

    Prometheus: 需要支持哪些支付方式？
你: 支付宝、微信支付、信用卡

Prometheus: 是否需要支付失败重试机制？
你: 是的，最多重试 3 次

Prometheus: 订单状态如何流转？
你: 待支付 → 已支付 → 发货中 → 已完成

  4. 确认计划

    Prometheus 生成详细计划并保存到 .sisyphus/plans/支付流程计划.md

    计划包含：

    • 任务列表
    • 验收标准
    • 风险评估
    • 技术选型建议

  5. 执行计划

核心特性

IntentGate — 意图门

真正行动前先分析用户真实意图，避免字面误解。

用户输入 → IntentGate 分析 → 真实意图 → 合适的 Agent
     "优化登录"        "重写登录组件"   Frontend Architect

功能：

  • 深入分析用户请求背后的真实需求
  • 区分表面需求 vs 深层需求
  • 避免字面理解导致的错误行动
  • 智能任务分类和路由

Hash-Anchored Edit Tool — 哈希锚定编辑

每行代码带有内容哈希标签，解决编辑失效问题：

优势：

  • 内容哈希验证每次修改
  • 零 stale-line 错误
  • 文件变更后自动拒绝编辑
  • 成功率从 6.7% 提升到 68.3%

工作原理：

  1. Agent 读取文件，每行获得哈希标签 (LINE#ID)
  2. 编辑时引用标签，验证内容是否匹配
  3. 哈希不匹配则拒绝编辑，重新读取
  4. 哈希匹配则安全应用修改

LSP + AST-Grep

功能命令用途
LSP 重命名lsp_rename跨文件符号重命名
LSP 跳转lsp_goto_definition跳转到定义
LSP 查找引用lsp_find_references查找所有引用
LSP 诊断lsp_diagnostics预构建错误检查
AST-Grepast_grep模式感知的代码搜索

支持 25+ 种主流编程语言。

Background Agents — 后台并行专家

同时发射多个专家 Agent 并行工作：

Sisyphus (主控)
    ├─→ Explore (代码搜索)
    ├─→ Librarian (文档查找)
    ├─→ Oracle (架构审查)
    ├─→ Hephaestus (代码实现)
    └─→ Metis (计划咨询)

优势：

  • 上下文保持精简
  • 结果准备好后汇总
  • 效率多倍提升

/init-deep — 深度初始化

在项目中生成层级化的 AGENTS.md 文件系统。

工作原理：

/init-deep 遍历项目目录结构，在每个关键层级创建 AGENTS.md 文件：

项目根目录 AGENTS.md → 项目整体架构、技术栈、全局规范
        ↓
src/AGENTS.md → 模块职责、依赖关系、编码约定
        ↓
src/components/AGENTS.md → 组件设计模式、通信约定

用法：

生成的文件结构：

my-project/
├── AGENTS.md              # 项目全局上下文
├── src/
│   ├── AGENTS.md          # 模块级上下文
│   └── components/
│       └── AGENTS.md      # 组件级上下文

何时使用：

适合新项目初始化、接手遗留项目、大型重构前、团队协作项目。

不适合简单单文件脚本、临时测试项目、个人熟悉的小项目。

不同项目类型策略：

  • 前端项目：/init-deep --max-depth=4，重点关注 components、lib、hooks
  • 后端项目：/init-deep --max-depth=5，重点关注 controllers、services、models、routes
  • Monorepo：/init-deep，为每个 package 单独生成 AGENTS.md

/ulw-loop — 持续工作循环

让 AI 持续工作直到任务 100% 完成。

/ralph-loop — 自引用循环

更智能的循环，AI 会自我评估和改进。

Agent 系统

主 Agent

Sisyphus — 主指挥官

属性描述
默认模型claude-opus-4-6 / kimi-k2.5 / glm-5
思考预算32,000 tokens
角色定位主编排者

特点：

  • 意图门：真正行动前先分析用户真实意图
  • 编排大师：调度专家团队
  • 激进并行：同时发射多个 Agent 并行工作
  • Todo 强制执行：确保任务必须完成
  • 绝不半途而废：任务完成前绝不停止

Hephaestus — 自主深度工作者

属性描述
默认模型gpt-5.3-codex
角色定位工匠
运行模式完全自主

特点：

  • 只需给目标，不给具体做法
  • 自动探索，理解代码库模式
  • 从头到尾独立执行
  • 专注于单一复杂任务

Prometheus — 战略规划师

属性描述
默认模型claude-opus-4-6 / kimi-k2.5 / glm-5
思考预算32,000+ tokens
角色定位规划者

特点：

  • 面试模式：采访用户，深挖需求
  • 生成工作计划：产出计划文件
  • 只规划不实现：系统层面禁止修改代码
  • 集成研究 Agent：自动调用专家收集信息

触发方式： /start-work 命令或按 [Tab] 键

Metis — 计划顾问

属性描述
默认模型claude-opus-4-6 / kimi-k2.5 / glm-5
角色定位计划顾问

特点：

  • 审查和优化执行计划
  • 迭代改进计划细节
  • 评估计划执行难度
  • 分析所需资源

调用： Prometheus 自动调用，或 @metis

子 Agent

属性描述

特点：

协作流程

用户请求复杂功能
    ↓
Sisyphus (主编排者)
  → 意图门：分析真实意图
  → 决定：需要规划，启动 Prometheus
    ↓
Prometheus (规划师)
  → 并行启动 Explore (搜代码模式)
  → 并行启动 Librarian (搜外部文档)
  → 咨询 Oracle (架构建议)
    ↓
Momus (审核员)
  → 审核计划：通过/拒绝
    ↓
Sisyphus/Hephaestus (执行)
  → 按计划实现代码

最佳实践

  1. 日常开发 → 直接用 ulw 命令
  2. 复杂新功能 → 先 /start-work 让 Prometheus 规划
  3. 研究性问题 → 并行启动 Explore + Librarian
  4. 困难 Bug → 先自己试，2 次失败后咨询 Oracle
  5. 架构决策 → 直接咨询 Oracle
