# Agent 设计落地检查清单

原书相关章节：

- [总目录](https://jimmysong.io/zh/book/agentic-design-patterns/)
- [结论](https://jimmysong.io/zh/book/agentic-design-patterns/conlusion/)
- [附录 G：编程智能体](https://jimmysong.io/zh/book/agentic-design-patterns/appendix-g-coding-agents/)

这份清单用于把“想做一个 Agent”转成可评审的系统设计。

## 1. 目标与边界

- 这个 Agent 要完成什么具体目标？
- 成功标准是什么？
- 它明确不做什么？
- 用户输入不完整时，是追问、假设，还是拒绝执行？
- 任务结束条件是什么？

## 2. 自主性等级

- 它只是给建议，还是会执行动作？
- 它能改写哪些外部状态？
- 哪些动作必须人工确认？
- 哪些动作允许自动重试？
- 哪些动作必须禁止？

## 3. 工作流结构

- 是否需要提示链？
- 是否需要任务路由？
- 是否有可并行子任务？
- 是否需要先规划再执行？
- 中间产物是否可检查？

## 4. 工具设计

- 工具参数是否结构化？
- 工具失败是否有明确错误码或错误类型？
- 工具调用是否有日志？
- 高风险动作是否专用化？
- 是否避免把不可逆操作藏在通用工具里？

## 5. 知识与上下文

- 需要哪些外部知识源？
- RAG 是否有召回、排序、压缩和引用？
- 哪些内容进入短期上下文？
- 哪些内容进入长期记忆？
- 记忆是否标注来源、时间和可信度？

## 6. 多智能体协作

- 为什么需要多个 Agent？
- 每个 Agent 的责任边界是什么？
- 谁拥有最终决策权？
- 写入范围是否冲突？
- 子 Agent 输出是否有统一格式？

## 7. 安全与护栏

- 输入是否需要风险分类？
- 工具权限是否最小化？
- 是否有提示注入防护？
- 是否有敏感信息泄露检查？
- 是否有人工升级路径？

## 8. 异常恢复

- 工具失败如何重试？
- 计划失败如何重规划？
- 验证失败如何回滚？
- 外部依赖不可用时如何降级？
- 用户目标变化时如何更新状态？

## 9. 评估与监控

- 离线评估集覆盖哪些任务？
- 有没有对抗样本和回归样本？
- 上线后监控哪些指标？
- 失败案例如何沉淀为测试？
- 成本和延迟是否可观测？

## 10. 最小可行版本

一个可靠的 MVP Agent 不需要覆盖所有模式，但至少应该具备：

- 明确目标。
- 有限工具。
- 可见计划。
- 关键动作确认。
- 可运行验证。
- 失败日志。
- 人工接管路径。

我的判断是：先做一个窄而可靠的 Agent，比做一个宽而不可控的 Agent 更有价值。Agent 能力应该随着边界、验证和治理一起扩张，而不是先把权限全部打开。
