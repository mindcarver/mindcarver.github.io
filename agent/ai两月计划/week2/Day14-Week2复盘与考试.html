Day 14：Week 2 复盘与考试

学习目标

过去六天你学了大量的工程技能：API 调用、结构化输出、Function Calling、Prompt 模板管理、后端架构设计、完整 Demo 的搭建。今天是 Week 2 的最后一天，要把这些散落的知识点整合成一个完整的知识体系，并通过一个综合实战来检验你真正掌握了多少。

复盘不是简单的回顾。它要求你站在更高的视角，看清各个知识点之间的关联，发现自己理解薄弱的地方，为 Week 3 的学习做好准备。

今天的学习目标有三个：通过复盘建立 Week 2 的完整知识框架，通过综合实战验证你能否独立完成一个完整的 AI 应用模块，通过总结明确 Week 3 需要重点关注的领域。

核心概念

一、API 调用复盘

Day 8 的核心是”怎么可靠地调用大模型 API”。这个问题看起来简单，实际上涉及密钥管理、请求构造、流式输出、参数调优、错误处理、重试策略、速率限制、日志记录等多个工程问题。

复盘时要问自己的关键问题：

你能不能不看资料，完整描述一次 API 调用的流程？从构造请求到处理响应，中间经过了哪些环节？

如果 API 调用突然失败，你能根据错误信息快速定位问题吗？401 是什么错误？429 是什么错误？5xx 是什么错误？每种错误应该怎么处理？

你的 LLM Client 封装是否考虑了扩展性？如果要支持 OpenAI 以外的模型（比如 Claude、DeepSeek），需要改多少代码？

API 调用的工程本质是”为不确定的模型服务建立确定性的调用边界”。模型可能慢、可能出错、可能拒绝服务，但你的调用层应该是稳定的——有超时保护、有重试兜底、有日志可查、有降级可用。

需要特别回顾的知识点：

指数退避重试的原理和参数选择。为什么每次等待时间要翻倍？为什么加抖动？最大重试次数怎么定？

Token 统计的意义。为什么每次调用都要记录 Token 消耗？Token 统计数据能用来做什么？

Streaming 的实现细节。SSE 协议的工作方式、首 Token 时间的计算、流式输出下的 Token 统计。

二、结构化输出复盘

Day 9 的核心是”让模型输出能被程序可靠处理的数据格式”。

结构化输出的工程本质是”在模型的概率生成和程序的确定性要求之间建立桥梁”。模型生成的是概率性的自然语言，程序需要的是确定性的结构化数据。JSON Mode、JSON Schema、Pydantic 校验、格式修复、自动重试——这些机制共同构成了这座桥梁。

复盘时要问自己的关键问题：

JSON Mode 和 JSON Schema 的区别是什么？一个保证格式合法，一个保证内容合规。两者为什么需要配合使用？

你设计 JSON Schema 的策略是什么？是从简单版本开始逐步增加复杂度，还是一开始就设计完整结构？哪种方式成功率更高？

当模型输出的 JSON 校验失败时，你的重试策略是什么？为什么”带反馈的重试”比无脑重试有效？

Pydantic 在结构化输出流程中扮演了什么角色？它和 JSON Schema 是什么关系？

需要特别回顾的知识点：

JSON Schema 的设计技巧。如何定义合理的字段类型？如何设置合适的枚举值？如何添加有用的描述？

格式修复的触发条件。什么样的格式错误可以自动修复？什么样的错误必须重试？

嵌套结构的处理策略。如何设计多层嵌套的 Schema？如何降低嵌套结构的失败率？

三、Function Calling 复盘

Day 10 的核心是”让模型能够调用外部工具”。

Function Calling 的工程本质是”把模型的推理能力和程序的执行能力连接起来”。模型负责判断什么时候需要调用工具、调用哪个工具、传什么参数。程序负责执行工具、获取结果、返回给模型。

复盘时要问自己的关键问题：

Function Calling 的工作流程你能完整复述吗？从用户输入到最终输出，中间经过了哪些步骤？

什么场景适合用 Function Calling？什么场景用普通 Prompt 就够了？不要为了用而用。

工具定义的细节。工具名称怎么定？描述怎么写？参数怎么设计？这些细节会影响模型的选择。

你处理过 Function Calling 的哪些边界情况？工具执行失败怎么办？工具返回的数据量太大怎么办？模型一直循环调用工具怎么办？

需要特别回顾的知识点：

工具 Schema 的设计原则。描述要清晰具体、参数类型要准确、必要参数和可选参数要区分。

多工具的场景处理。模型如何从多个工具中选择合适的？工具之间有依赖关系怎么处理？

工具结果的返回格式。模型能理解什么样的工具返回格式？应该做多少数据预处理？

四、Prompt 模板复盘

Day 11 的核心是”系统化管理你的 Prompt”。

Prompt 模板管理的工程本质是”把经验固化为可复用的资产”。一个好的 Prompt 是经过多次迭代调优的结果。如果不系统化管理，每次都从头写 Prompt，就浪费了之前的经验积累。

复盘时要问自己的关键问题：

你的 Prompt 模板是怎么分类的？按功能分类？按业务场景分类？分类依据是什么？

模板的版本管理怎么做？怎么知道某个功能正在使用哪个版本的模板？如何快速回滚到旧版本？

模板的效果如何评估？人工评估？自动化评估？评估数据如何驱动模板优化？

模板的参数设计有哪些原则？参数名怎么定？默认值怎么设？如何避免参数过多导致模板难以使用？

需要特别回顾的知识点：

模板命名的规范。一个好的模板名应该包含哪些信息？如何避免命名冲突？

模板测试用例的设计。每个模板应该有几个测试用例？测试用例应该覆盖哪些场景？

模板评分的维度。准确性、稳定性、成本。每个维度如何量化？

五、后端架构复盘

Day 12 的核心是”把 AI 能力组织成一个可维护的后端系统”。

后端架构的工程本质是”为 AI 能力提供可靠的交付渠道”。模型能力再强，如果后端不稳定、不可扩展、难以维护，就很难在实际业务中使用。

复盘时要问自己的关键问题：

路由设计遵循什么原则？RESTful 风格？为什么有些接口不严格遵循 REST？

服务层怎么拆分？按业务领域拆分？按技术职责拆分？拆分的粒度如何把握？

配置管理的最佳实践是什么？配置和代码如何分离？不同环境的配置如何管理？

日志系统包含哪些信息？哪些信息是必须的？哪些信息是可选的？如何在详细程度和性能之间平衡？

错误处理怎么设计？错误分类的标准是什么？错误信息应该如何返回给用户？

需要特别回顾的知识点：

FastAPI 的核心概念。路由、依赖注入、请求体、响应模型、中间件——每个概念在 AI 应用后端中的典型用法。

项目目录的组织方式。api 层、service 层、core 层各负责什么？如何组织让代码易于维护和扩展？

中间件的使用场景。日志中间件、错误处理中间件、CORS 中间件——每个中间件的作用和实现方式。

六、完整 Demo 复盘

Day 13 的核心是”把所有知识串起来”。

完整 Demo 的工程本质是”验证各个模块能够协同工作”。单独学习每个知识点时，它们是孤立的。只有在一个完整的系统中，你才能看到它们之间的关联和配合。

复盘时要问自己的关键问题：

输入规范化为什么重要？不做规范化会出什么问题？

任务解析的逻辑是怎样的？如何把用户模糊的需求转化为具体的分析维度？

Prompt 模板选择的依据是什么？不同行业、不同深度应该用不同的模板吗？

结构化输出在 Demo 中扮演什么角色？如果不用结构化输出，会有什么问题？

报告生成是必要的吗？直接返回 JSON 数据不行吗？

结果保存有什么价值？缓存、历史记录、质量追踪——哪个价值最大？

完整链路中的哪个环节最容易出错？如何提高整体稳定性？

需要特别回顾的知识点：

完整链路的七个环节。输入规范化→任务解析→模板选择→模型调用→结构化输出→报告生成→结果保存。

每个环节可能的错误类型。输入为空、模型超时、输出格式错误、数据库连接失败。

错误处理的设计思路。哪个环节的错误可以自动恢复？哪个环节的错误需要返回给用户？

学习建议

动手实践。本周的代码都要亲自运行一遍。不是看懂就行，是要能自己写出来。

关注边界情况。不仅要考虑正常流程，还要考虑各种异常情况。输入为空、超时、失败、格式错误——这些情况怎么处理？

性能意识。Token 消耗、响应时间、并发能力——这些都是生产环境中需要关注的问题。

代码质量。保持代码清晰、可维护。变量命名、函数拆分、错误处理、日志记录——这些细节决定了代码的可维护性。

阅读文档。多读官方文档，了解最新的特性和最佳实践。AI 领域变化很快，官方文档是最可靠的信息来源。
