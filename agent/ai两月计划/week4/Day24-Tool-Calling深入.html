Day 24：Tool Calling 深入

学习目标

昨天建立了 Agent 的基础认知，今天要深入 Agent 的”手和脚”：Tool Calling。

Tool Calling 是让大模型从”只会说话”进化到”能做事”的关键技术。没有 Tool Calling，大模型只能生成文本；有了 Tool Calling，大模型能调用外部工具、查询数据库、执行计算、发送消息，真正参与到业务流程中去。

今天的目标是深入理解 Tool Calling 的每个环节：工具怎么定义、模型怎么选择工具、参数怎么传递、结果怎么返回、失败了怎么办、权限怎么控制、怎么审计。学完这一天，你应该能为一个行业研究 Agent 设计出完整的工具集，并定义每个工具的 Schema、权限级别和调用规范。

核心概念

Tool Schema

Tool Schema 是工具的”身份证”。它告诉模型：这个工具叫什么、能做什么、需要什么参数、返回什么格式。

一个完整的 Tool Schema 包含四个部分。

第一部分是工具名称（name）。名称要简洁、有意义、符合命名规范。好的命名让模型一看就知道这个工具做什么。比如 search_web 比 tool_1 好，calculate_roi 比 compute 好。命名建议用”动词_名词”格式，全小写，下划线分隔。

第二部分是工具描述（description）。描述要准确、具体、无歧义。这是模型选择工具的主要依据。差的描述：“搜索信息”。好的描述：“根据关键词搜索互联网上的公开信息，返回搜索结果的标题、摘要和链接。适用于需要获取最新资讯、查找公开数据、了解行业动态的场景。不适用于查询内部数据库或私有知识库。”

描述中建议包含三块内容：这个工具做什么、适用什么场景、不适用什么场景。特别是”不适用场景”的说明，能有效减少模型误选工具的情况。

第三部分是输入参数（parameters）。用 JSON Schema 格式定义，包含每个参数的名称、类型、描述、是否必填。参数设计的原则是”最少必要参数”——不要让模型传它不需要的参数，也不要遗漏关键参数。

第四部分是输出格式（returns）。定义工具返回数据的结构。输出格式要稳定、可预测，不要这次返回一个结构下次返回另一个结构。

举个具体例子。search_web 工具的完整 Schema：

名称：search_web 描述：根据关键词搜索互联网上的公开信息，返回搜索结果的标题、摘要和链接 参数：

  • query（string，必填）：搜索关键词，建议用简短的名词短语
  • max_results（integer，可选）：返回结果数量，默认 5，范围 1-10
  • language（string，可选）：搜索语言，默认 zh-CN

输出：

  • results（array）：搜索结果列表
    • title（string）：结果标题
    • snippet（string）：内容摘要
    • url（string）：链接地址
    • source（string）：来源网站

设计 Tool Schema 时最常犯的错误是描述太简略。很多人写一句”搜索工具”就完事了，结果模型不知道什么时候该用这个工具、什么时候不该用。花时间把描述写好，是 Tool Calling 成功的一半。

工具选择

模型怎么知道该用哪个工具？它基于两个信息：工具描述和当前任务上下文。

工具描述已经讲过了，它是模型选择工具的主要依据。另一个关键信息是当前任务上下文——Agent 现在在做什么、目标是什么、已经执行了哪些操作、当前状态是什么。

工具选择的过程本质上是推理。模型会问自己：“我当前要达成什么目标？有哪些工具可用？每个工具能做什么？哪个工具最能帮我接近目标？”

举个例子。Agent 的当前目标是”获取半导体行业的市场规模数据”。可用工具有 search_web、query_database、calculate_metrics。模型推理：市场规模数据是行业数据，可能在网上找到，search_web 能搜索互联网信息，应该用它。数据库可能有历史数据但不一定有最新的，query_database 不作为首选。计算工具不相关。最终选择 search_web。

模型也可能选错。比如 Agent 要查询公司内部的文档，但模型选择了 search_web 而不是 query_knowledge_base。这种错误通常有两个原因：一是工具描述不够清晰，模型不知道 query_knowledge_base 能做什么；二是模型对当前任务的理解有偏差。

参数传递

模型选定工具后，需要生成调用参数。参数生成是一个推理过程：模型根据工具的参数定义和当前任务需求，推断每个参数应该填什么值。

参数传递有两种方式。一种是模型直接生成参数值。比如调用 search_web，模型根据当前任务”查找半导体行业”，推断 query 参数应该是”半导体行业市场规模”。另一种是系统自动填充部分参数。比如 user_id、session_id 这类上下文参数，系统可以直接从会话信息中提取，不需要模型生成。

参数传递的常见问题：

参数类型错误。工具要求 integer，模型生成了 string。这会导致工具调用失败。参数类型要在 Schema 中明确定义，并在描述中举例说明。

参数格式错误。日期参数要求 YYYY-MM-DD 格式，模型生成了”2024年1月1日”。这需要在参数描述中明确格式要求。

必填参数缺失。模型没有提供某个必填参数的值。这通常是因为模型认为这个参数不重要，或者从上下文中无法推断。解决方法是在描述中强调”该参数为必填项”。

参数值不合理。max_results 参数范围是 1-10，模型生成了 100。这需要在参数描述中明确约束，或者在工具实现中做参数校验和修正。

结果返回

工具执行后需要返回结果给模型。结果返回的设计要考虑三件事：返回什么格式、返回多少数据、怎么处理错误。

返回格式应该和 Tool Schema 的 returns 定义一致。如果 Schema 说返回 {results: [{title, snippet, url, source}]}，那工具实现就必须按这个格式返回。格式不一致会导致模型解析失败。

返回数据量要控制。一次工具调用返回 1000 条结果，模型处理不过来，而且容易超出上下文窗口。建议在工具实现中做数据裁剪，比如只返回最相关的 top-10 或 top-20。

错误处理是关键。工具可能因为各种原因失败：网络超时、API 限流、参数错误、权限不足。错误信息要清晰，告诉模型”什么错了”和”为什么错”，而不是返回一个笼统的”调用失败”。

错误返回格式建议：

{
  "success": false,
  "error": "API rate limit exceeded",
  "error_code": "RATE_LIMIT",
  "retry_after": 60
}

这样模型能理解错误的性质，决定是重试、等待还是换一个策略。

工具权限

不是所有 Agent 都能调用所有工具。工具权限控制是 Agent 安全的基础。

权限控制有三个层次。

工具白名单。每个 Agent 只能调用白名单内的工具。白名单之外的工具调用请求会被拒绝。比如”行业研究 Agent”只能调用 search_web、query_knowledge_base、calculate_metrics，不能调用 send_email、delete_file。

参数限制。即使工具在白名单中，也要限制参数范围。比如 search_web 的 query 参数限制长度（不超过 200 字符），query_database 的 table_name 参数限制在允许的表列表内，send_email 的 to 参数限制在企业邮箱域名内。

操作审批。高风险工具（发送邮件、创建任务、写入文件、删除数据）的每次调用都需要人工审批。这在 Day 26 的 Human-in-the-loop 中会详细讲。

工具审计

生产环境中需要记录所有工具调用，用于审计和调试。

审计日志应该包含：调用时间、Agent ID、工具名称、参数、执行结果、耗时、错误信息。这些日志能回答”谁在什么时候调用了什么工具、传了什么参数、结果如何、有没有出错”。

审计日志有两个用途。一是调试。当 Agent 行为异常时，通过审计日志追溯调用历史，定位问题。二是合规。企业级应用需要知道 AI 系统做了什么操作，特别是涉及外部系统或敏感数据的操作。

审计日志要安全存储。防止篡改、控制访问权限、设置保留期限。日志中可能包含敏感信息，需要脱敏处理。

工具设计最佳实践

总结一下设计工具时要注意的关键点：

名称用”动词_名词”格式，简洁有意义。

描述包含三部分：做什么、适用场景、不适用场景。描述越详细，模型选择越准确。

参数遵循”最少必要”原则，避免让模型传不需要的参数。参数类型、格式、约束要在描述中明确说明。

输出格式稳定，和 Schema 定义一致。控制返回数据量，避免返回过多数据。

错误信息清晰，告诉模型”什么错了”和”为什么错”。

权限控制用白名单、参数限制、操作审批三层防护。

审计日志记录所有调用，用于调试和合规。

今日总结

Tool Schema 是工具的”身份证”，包含名称、描述、参数、输出四部分。

工具选择基于工具描述和当前任务上下文，本质上是模型推理。

参数传递要注意类型、格式、必填项、合理性。

结果返回要控制格式、数据量、错误处理。

权限控制三个层次：工具白名单、参数限制、操作审批。

审计日志记录所有调用，用于调试和合规。

工具设计的关键：描述要详细、参数要最少、格式要稳定、错误要清晰。

明日预告

今天学了单 Agent 的工具调用。单 Agent 能处理很多任务，但当任务特别复杂、需要多个领域的专业知识时，单 Agent 会力不从心。明天要讲的是 Multi-Agent——让多个专业 Agent 协作完成复杂任务。
