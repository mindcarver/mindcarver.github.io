# 记忆、学习与反思：Agent 如何持续改进

## 1. 反思：把失败变成下一轮输入

反思模式让 Agent 在输出后检查自己的结果，发现遗漏、矛盾、风险或不满足要求的地方，再进入修正。

一个典型反思循环是：

```text
生成初稿
  -> 按标准审查
  -> 找出缺陷
  -> 修改
  -> 再审查
  -> 达到停止条件
```

反思的价值在于提高复杂任务质量，尤其适合写作、代码生成、计划制定、需求分析和多步骤推理。

但反思也有风险：如果没有外部证据，它可能只是在“自圆其说”；如果没有停止条件，它可能无限打磨。反思最好与测试、检索、用户反馈或规则校验结合，而不是完全依赖模型自评。

## 2. 记忆：让 Agent 不必每次从零开始

记忆管理解决两个问题：当前任务中如何保留必要状态，跨会话如何复用历史知识。

可以把 Agent 记忆分成四类：

- **工作记忆**：当前任务计划、已做步骤、临时结论。
- **情景记忆**：某次任务发生了什么、为什么这样决策。
- **语义记忆**：抽象后的规则、偏好、项目知识。
- **程序记忆**：可复用流程、检查清单、操作习惯。

不是所有历史都值得记住。记忆系统必须有筛选和遗忘机制。否则长期记忆会变成污染源：旧事实、错误结论、过期偏好会不断影响新任务。

## 3. 学习与适应：从单次反馈到策略更新

学习与适应比记忆更进一步。记忆保存事实，学习改变策略。

例如：

- 某类任务总在测试阶段失败，Agent 应该提前补测试。
- 某个用户总希望中文回答，Agent 应该自动适配语言。
- 某个仓库禁止新增依赖，Agent 应该把它变成默认约束。
- 某类 API 经常超时，Agent 应该选择更保守的调用方式。

这类适应不能随意发生。否则一次偶然失败就可能被过度泛化为长期规则。更可靠的做法是把学习分层：

- 单次任务内：作为临时注意事项。
- 多次重复后：沉淀为项目经验。
- 人工确认后：进入长期规则或技能。

## 4. 反思、记忆、学习的边界

三者经常被混在一起，但职责不同：

- 反思关注本次输出是否足够好。
- 记忆关注哪些信息下次还要用。
- 学习关注未来行为策略是否要调整。

一个安全的循环应该是：

```text
执行任务
  -> 验证结果
  -> 反思失败原因
  -> 记录必要证据
  -> 提炼可复用规则
  -> 在合适边界内更新行为
```

## 5. 我的理解

Agent 的长期能力不是靠无限上下文堆出来的，而是靠“可治理的记忆”形成的。

好的记忆系统应该保守：宁可少记，也不要乱记；宁可保存证据，也不要只保存结论；宁可标注时间和来源，也不要把过去判断伪装成永恒事实。

对工程 Agent 来说，最有价值的记忆不是“用户说过什么”，而是“这个项目真实约束是什么、哪些路径试过失败、哪些验证命令可信”。
