1.1 从多合约到单例

PoolId设计原理：

flowchart LR
    Input[PoolKey] --> Encode[abi.encode]

为什么使用PoolId而不是合约地址？

对比项V3（合约地址）V4（PoolId）
存储位置独立合约mapping索引
部署成本CREATE2 (~100k gas)纯计算 (0 gas)
查询成本EXTERNALCALL (2,600 gas)SLOAD (2,100 gas)
批量操作需要多笔交易单笔交易完成
状态同步需要跨合约调用直接访问mapping

3. 瞬时会计系统

3.1 设计理念

传统模式（V3）的问题：

sequenceDiagram
    participant U as User
    participant P as V3 Pool
    participant T as Token

    U->>P: swap(amountIn)
    Note over P,T: 每个操作都需要<br/>实际的代币转账
    P->>T: transfer(user, amountOut)
    P->>U: callback()
    U->>T: transfer(pool, amountIn)
    P->>P: 验证余额

    Note over U,T: 多次外部调用<br/>高gas成本<br/>无法批量操作

瞬时会计模式（V4）的优势：

sequenceDiagram
    participant U as User
    participant P as V4 PoolManager
    participant T as Token

    U->>P: swap(amountIn)
    Note over P: 仅记录账面变化<br/>无实际转账
    P->>P: deltas[user][token] += amountOut
    P->>P: deltas[user][token] -= amountIn
    P-->>U: 返回结果

    Note over U,T: 结算阶段<br/>（unlock时）
    P->>T: transferFrom(user, pool, amountIn)
    P->>T: transfer(user, amountOut)

    Note over U,T: 单次结算<br/>可批量操作<br/>Gas优化

6.2 瞬时会计总结

mindmap
  root((瞬时会计))
    核心概念
      延迟转账
      账面记录
      最终结算
    deltas映射
      正数=应收
      负数=应付
      账户级追踪
    settle函数
      声明应付款
      验证msg.value
      记录差额
    take函数
      声明应收款
      unlock时转账
    批量操作
      累积差额
      一次性结算
