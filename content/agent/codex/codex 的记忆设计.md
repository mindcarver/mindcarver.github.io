# Codex CLI 记忆系统设计详解

> Codex CLI 是 OpenAI 推出的轻量级终端 AI 编程助手，其记忆系统（v0.100.0+）是其实现长期上下文保持的核心能力。

---

## 一、设计原理

### 1.1 核心问题：AI 代理的"失忆"

传统 AI 编程助手存在一个根本性缺陷：**每次会话都是全新的开始**。这导致：

- ❌ 重复解释项目架构和业务逻辑
- ❌ 编码偏好和规范无法跨会话保持
- ❌ 重要架构决策需要反复说明
- ❌ 长期项目缺乏连贯的开发上下文

### 1.2 解决思路：外部化记忆系统

Codex CLI 的记忆系统采用 **外部化持久化存储** 的设计思路：

```
┌─────────────────────────────────────────────────────────────┐
│                    Codex 记忆系统架构                         │
├─────────────────────────────────────────────────────────────┤
│                                                              │
│  ┌──────────────┐     ┌──────────────┐     ┌──────────────┐ │
│  │  会话启动    │ ──▶ │  记忆加载    │ ──▶ │  上下文注入   │ │
│  └──────────────┘     └──────────────┘     └──────────────┘ │
│         │                    │                    │        │
│         ▼                    ▼                    ▼        │
│  ┌──────────────┐     ┌──────────────┐     ┌──────────────┐ │
│  │  工作执行    │ ──▶ │  记忆更新    │ ──▶ │  持久化存储   │ │
│  └──────────────┘     └──────────────┘     └──────────────┘ │
│                                                              │
│  ┌─────────────────────────────────────────────────────┐   │
│  │           .codex/ 或项目 memory-bank/               │   │
│  │  ├─ facts.md          (项目事实)                    │   │
│  │  ├─ preferences.md    (编码偏好)                    │   │
│  │  ├─ projectbrief.md   (项目概要)                    │   │
│  │  ├─ activeContext.md  (当前上下文)                  │   │
│  │  └─ progress.md       (进度摘要)                    │   │
│  └─────────────────────────────────────────────────────┘   │
└─────────────────────────────────────────────────────────────┘
```

**核心设计原则：**

| 原则 | 说明 |
|------|------|
| **分离存储** | 记忆独立于模型运行时，存储在本地文件系统 |
| **结构化组织** | 使用分层 Markdown 文件，按信息类型分类 |
| **增量更新** | 每次会话结束后选择性更新相关文件 |
| **会话独立性** | 记忆在会话之间持久存在，不依赖对话历史 |
| **轻量高效** | 只在会话开始时加载一次，避免上下文膨胀 |

### 1.3 记忆层次结构

Codex CLI 的记忆系统采用 **三层金字塔结构**：

```
                    ┌──────────────────┐
                    │   项目概要       │
                    │  projectbrief.md │
                    │   (静态不变)     │
                    └────────┬─────────┘
                             │
           ┌─────────────────┴─────────────────┐
           │                                   │
    ┌──────▼──────┐                     ┌──────▼──────┐
    │   事实层     │                     │   偏好层     │
    │  facts.md   │                     │preferences.md│
    │ (累积增长)   │                     │ (相对稳定)    │
    └──────┬──────┘                     └──────┬──────┘
           │                                   │
           └─────────────────┬─────────────────┘
                             │
                    ┌────────▼────────┐
                    │   动态上下文     │
                    │activeContext.md │
                    │  (频繁更新)      │
                    └─────────────────┘
```

## 二、文件结构详解

### 2.1 核心文件

| 文件 | 用途 | 更新频率 | 大小建议 |
|------|------|----------|----------|
| **`projectbrief.md`** | 项目目标、范围、核心概念 | 初始化后很少变化 | 1 页以内 |
| **`facts.md`** | 架构事实、技术栈、关键决策 | 增量累积 | 精简条目 |
| **`preferences.md`** | 编码风格、工具偏好、规范 | 相对稳定 | 具体明确 |
| **`activeContext.md`** | 当前工作状态、进行中任务 | 每会话更新 | 仅当前状态 |
| **`progress.md`** | 里程碑摘要、已完成功能 | 定期更新 | 摘要式 |

### 2.2 文件内容示例

#### `projectbrief.md`
```markdown
# 项目概要

## 目标
构建一个实时协作的在线白板工具，支持多人同时编辑和图形绘制。

## 核心功能
- WebSocket 实时同步
- Canvas 2D 绘图引擎
- 多用户光标跟踪
- 导出 PNG/PDF

## 技术栈
- 前端: React + TypeScript
- 后端: Node.js + Socket.io
- 存储: PostgreSQL
```

#### `facts.md`
```markdown
# 项目事实

## 架构决策
- 使用 CRDT 算法处理冲突解决（2024-01-15）
- Canvas 渲染层与逻辑层分离（性能优化）
- 状态管理采用 Zustand（避免 Redux 冗余）

## 重要约定
- API 路由前缀统一为 `/api/v1/`
- 组件使用 PascalCase 命名
- 工具函数使用 camelCase 命名
```

#### `preferences.md`
```markdown
# 编码偏好

## 风格规范
- 使用 2 空格缩进
- 单引号优先
- 箭头函数用于回调，class 方法用于对象方法

## 工具配置
- ESLint: @typescript-eslint/recommended
- Prettier: 单行最大 80 字符
- 测试框架: Vitest

## Git 工作流
- 功能分支命名: `feature/描述`
- 提交信息: Conventional Commits
```

#### `activeContext.md`
```markdown
# 当前上下文

## 进行中任务
- [ ] 实现用户权限系统（当前 50%）
- [ ] 添加撤销/重做功能（阻塞中）

## 当前分支
`feature/user-permissions`

## 最近变更
- 添加了 RBAC 权限模型
- 实现了中间件权限检查

## 已知问题
- WebSocket 连接在高并发下不稳定
- 导出 PDF 时中文乱码
```

## 三、使用场景

### 3.1 场景一：长期项目维护

**问题场景：**
一个持续 6 个月的项目，每周多次开发，但每次重新打开项目都需要重新解释架构。

**记忆系统解决方案：**

```
第1周：初始化 memory-bank
├─ 创建 projectbrief.md（项目目标和范围）
├─ 记录初始技术选型到 facts.md
└─ 定义编码规范到 preferences.md

第4周：架构调整
└─ 更新 facts.md：添加"从 Redux 迁移到 Zustand"

第12周：新人加入
└─ 新成员通过 projectbrief.md 快速了解项目

第24周：功能迭代
└─ activeContext.md 记录当前冲刺任务
```

**效果：**
- ✅ 新成员 30 分钟理解项目（vs 传统 2 小时）
- ✅ 编码风格自动保持一致
- ✅ 架构决策有完整历史记录

### 3.2 场景二：多开发者协作

**问题场景：**
团队多人使用 Codex CLI，每个人有不同的编码偏好，容易产生风格冲突。

**记忆系统解决方案：**

| 开发者 | 偏好 | 记忆文件配置 |
|--------|------|-------------|
| Alice | 函数式编程偏好 | `preferences.md` 中记录函数式模式 |
| Bob | OOP 偏好 | `preferences.md` 中记录类设计模式 |
| 团队共识 | 统一规范 | `preferences.md` 中定义团队标准 |

**效果：**
- ✅ Codex 自动遵循团队编码规范
- ✅ 减少代码审查中的风格争议
- ✅ 新人自动继承团队最佳实践

### 3.3 场景三：上下文切换

**问题场景：**
同时维护多个项目，频繁切换导致上下文混乱。

**记忆系统解决方案：**

```
~/projects/
├─ project-a/.codex/
│  └─ facts.md (Project A 技术栈)
├─ project-b/.codex/
│  └─ facts.md (Project B 技术栈)
└─ project-c/.codex/
   └─ facts.md (Project C 技术栈)
```

**效果：**
- ✅ 项目切换时自动加载对应记忆
- ✅ 避免跨项目的技术栈混淆
- ✅ 每个项目保持独立的开发上下文

### 3.4 场景四：复杂功能开发

**问题场景：**
开发一个跨多个模块的复杂功能，需要保持整体设计的一致性。

**记忆系统解决方案：**

```markdown
# activeContext.md (功能开发期间)

## 当前功能：用户认证系统

### 涉及模块
- `auth/` - 认证逻辑
- `api/middleware/` - JWT 验证中间件
- `frontend/components/` - 登录 UI

### 设计决策
- 使用 JWT 而非 Session（无状态需求）
- Token 有效期 24 小时
- Refresh Token 存储在 HttpOnly Cookie

### 待完成
- [ ] 密码重置流程
- [ ] 双因素认证（2FA）
```

**效果：**
- ✅ 跨模块的一致性自动维护
- ✅ 设计决策可追溯
- ✅ 中断后可快速恢复进度

## 四、记忆管理命令

Codex CLI v0.100.0+ 提供了记忆管理斜杠命令：

| 命令 | 功能 | 使用场景 |
|------|------|----------|
| `/m_update` | 手动触发记忆更新 | 完成重要功能后 |
| `/m_drop` | 清理过时记忆 | 项目方向调整时 |
| `/m_view` | 查看当前记忆 | 检查记忆状态 |

### 最佳实践

```bash
# 完成功能开发后
> /m_update
# Codex 将更新 activeContext.md 和 progress.md

# 项目架构重大调整
> 手动编辑 facts.md，删除旧架构决策
> /m_update
# Codex 将重新加载更新后的记忆

# 查看当前记忆状态
> /m_view facts.md
# 预览 facts.md 的内容
```

## 五、设计哲学

### 5.1 简洁性原则

> "每个文件都在会话开始时加载到上下文中，因此必须保持简洁。"

- ❌ **错误做法**：`progress.md` 记录每日详细变更日志
- ✅ **正确做法**：`progress.md` 只保留里程碑摘要

### 5.2 层次化原则

> "文件之间应该有清晰的层次关系，避免重复。"

```
projectbrief.md (是什么)
       ↓
facts.md (怎么做)
       ↓
activeContext.md (现在做什么)
```

### 5.3 可演进原则

> "记忆结构应该随着项目发展而演进。"

- 初期：只需 `projectbrief.md` 和 `facts.md`
- 中期：添加 `activeContext.md` 跟踪进度
- 成熟期：细分为组件级记忆（`memory-bank/components/`）

## 七、实战建议

### 7.1 项目启动时

```bash
# 1. 初始化 Codex 项目
codex init

# 2. 创建 memory-bank 目录
mkdir -p .codex

# 3. 让 Codex 帮助创建初始记忆
codex "请根据当前项目创建 memory-bank 结构，包括 projectbrief.md 和 facts.md"
```

### 7.2 日常开发中

```bash
# 完成一个重要功能后
codex "更新 activeContext.md：用户认证模块已完成"

# 项目方向调整时
codex "清理 facts.md 中关于旧架构的记录"

# 定期维护
codex "更新 progress.md：v1.2 版本已发布的所有功能"
```

### 7.3 团队协作

```bash
# 将 .codex/ 加入版本控制
git add .codex/
git commit -m "docs: 添加项目记忆系统"

# 团队成员首次拉取
git pull
# Codex 将自动加载项目的记忆文件
```

## 八、总结

Codex CLI 的记忆系统通过 **外部化、结构化、分层化** 的设计，解决了 AI 编程助手的"失忆"问题。它让 Codex 从一个"单次对话工具"转变为"长期开发伙伴"，特别适合：

- 🔄 **长期项目**：跨月、跨年的持续开发
- 👥 **团队协作**：多人共享编码偏好和架构决策
- 🔀 **多项目切换**：保持每个项目的独立上下文
- 📈 **复杂功能**：跨模块、跨文件的一致性维护

**核心价值：** 用极简的 Markdown 文件，实现 AI 记忆的持久化和共享化，让 AI 真正成为"有记忆"的开发伙伴。
