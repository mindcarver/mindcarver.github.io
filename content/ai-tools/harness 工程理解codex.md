# Harness 工程：OpenAI 的零人工编码实验

2025年8月，OpenAI团队开始了一个实验：构建一个软件产品，但**不允许任何人写一行代码**。这就是Harness工程。

他们花了5个月，做出了约100万行代码。整个过程中，人类工程师不直接写代码——他们设计环境、定义约束，然后让Codex智能体去做具体执行。

## 基本数据

- 开发周期：5个月
- 代码量：约100万行
- 团队：3名工程师（后来扩展到7人）
- Pull Request数量：约1500个
- 平均每人每天处理3.5个PR
- 效率：大概是手工编写所需时间的1/10

## 工作原理

### 智能体做开发

从初始化代码仓库到部署，全程由Codex完成。就连AGENTS.md这个告诉智能体怎么工作的文件，也是Codex自己写的。

团队有一条铁律：人类不写代码。

### 工程师改行

既然不能写代码，工程师就干别的事：

之前：写代码、调试、优化
现在：设计系统、定义规范、搭建工具、建立反馈机制

简单说，人类负责设计环境和规则，智能体负责把活干完。

## 几个关键做法

### 1. 情境管理：给地图，不给百科全书

Codex的情境窗口有限。如果AGENTS.md有1000页，任务、代码和文档的情境就被挤掉了。

所以他们只给了一张"地图"——大约100行的AGENTS.md，当作目录。结构化的docs/目录才是真正的知识库。

智能体从小切入点开始，需要时再被指导下一步去哪里看。linter和CI作业会验证知识库的更新是否正确。

### 2. 架构要严格

智能体在边界清晰、结构可预测的环境中最高效。

他们定义了分层依赖规则：每个业务域内只能"向前"依赖
```
Types → Config → Repo → Service → Runtime → UI
```

横切关注点（认证、遥测这些）只能通过Providers接口进入。其他方式一概不允许。

自定义linter（也是Codex生成的）会强制执行这些边界规则。

### 3. 定期清理

智能体会模仿代码库里的模式——包括坏模式。

所以团队制定了"黄金原则"：一些带主观判断的机械规则，用来保持代码库的一致性和可读性。

后台Codex任务会定期扫描偏差，自动重构代码。大多数清理工作一分钟就能审查完并自动合并。

这就像垃圾回收。技术债务是高利贷，小额定期还比累积一次还更划算。

### 4. 给智能体建工具

人工QA成了瓶颈，所以得给智能体造工具。

UI测试方面，应用程序基于git worktree启动。Codex可以为每次改动启动实例、接入Chrome DevTools协议、处理DOM快照和截图。

可观测性方面，本地堆栈让Codex用LogQL查日志、用PromQL查指标。这样复杂的提示变得可行——比如"确保服务启动在800ms内完成"。

## 工作流程

典型流程是这样的：

1. 人类描述任务
2. 运行智能体
3. Codex开PR
4. 智能体互相审核（本地审核、请求其他智能体审查、响应反馈循环）
5. 人类审核（可选，不是必须）
6. 合并

现在给定一个提示，智能体可以自己完成：
- 验证代码库状态、复现bug、录问题视频
- 实施修复、运行验证、录解决方案视频
- 开PR、回应反馈、修复构建故障
- 只在需要判断时才找人类
- 合并更改

## 代码库的特点

### 为智能体可读性优化

代码库是智能体写的，所以首先得让智能体能读懂：

情境本地化（把情境推送到仓库）、版本化工件（代码、Markdown、schema、可执行计划）、优先选可组合性强的依赖项。

### 技术选择偏好

选"枯燥"的技术——在训练集里表现好、API稳定、可组合性强的抽象。

比如他们没引入通用的p-limit包，而是写了自己的并发map辅助函数。与OpenTelemetry仪表紧密集成、100%测试覆盖率、行为完全符合运行时预期。

### 不变量强制执行

原则是强制执行不变量，而不是管实现过程。

比如要求在边界处解析数据形状，但不规定具体实现（模型偏好Zod但不强制）、结构化日志、schema和类型命名约定、文件大小限制。

## 工程规范要改

Codex吞吐量太大，传统工程规范跟不上。

合并门要尽量少。Pull Request生命周期很短。测试偶发失败就重跑，不能无限期卡着。

在智能体产出远超人类注意力的系统里，纠错成本低（快速发现修复），等待成本高（卡住进展代价更大）。

## 智能体写了什么

"代码库由Codex生成"指的是整个代码库：

产品代码和测试、CI配置和发布工具、内部开发者工具、文档和设计历史、评估框架、审阅评论和回复、管理代码仓库的脚本、生产仪表板定义文件。

人类还是参与的，只是层次不同：排优先级、转用户反馈为验收标准、验证结果、识别缺失的东西（工具、指导、约束、文档）、把缺失内容反馈回代码仓库。

## 当前的问题

还有几个挑战没解决：

1. 完全由智能体生成的系统，架构连贯性怎么随时间演变
2. 怎么把人类判断力编码进去，让它发挥更大作用
3. 模型能力不断增强，系统会怎么演变

但有一点是明确的：构建软件还是需要纪律。只是纪律更多体现在支撑结构（工具、抽象、反馈回路）、环境设计、控制系统——而不是代码本身。

## 对行业的启示

对工程团队：从编码者转向架构师，情境管理是稀缺资源，需要强化约束机制，建立快速反馈回路。

对工具开发：工具要让智能体直接用和理解，可观测性优先（让智能体能直接读和推理系统状态），自动化验证强制执行不变量。

对组织管理：重新定义工程流程，人类时间和注意力是真正稀缺的资源，建立人和智能体的新协作模式。

## 以后会怎样

智能体在软件生命周期中占的比重越来越大。

设计环境、反馈回路和控制系统变得更重要。大规模构建和维护复杂可靠的软件需要新方法论。人和智能体的协作模式会继续演进。

Harness工程提供了一个未来的样本：当软件工程团队不再以写代码为主要工作时，可能发生什么变化。

说实话，凌晨三点没人看着的时候，智能体还在不停地生成代码、开PR、修bug、合并——这画面有点超现实。开发社区一半人疯了，另一半人在解释为什么这不算数。真相可能在无聊的中间某处。但不管你怎么看，3个人5个月做出100万行可运行代码，这事确实发生了。

## 关键要点

1. 人类掌舵，智能体执行——不手动写代码
2. 情境是稀缺资源——给地图不给百科全书
3. 严格约束创造自由——通过边界和不变量强制执行
4. 工具比代码重要——投资支撑结构和反馈回路
5. 黄金原则持续应用——循环清理保持代码库质量
6. 为智能体可读性优化——让代码库本身成为知识源
