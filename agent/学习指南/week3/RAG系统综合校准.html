RAG 系统综合校准

文章导读

Week 3 的前六天，我们从 RAG 的原理一路学到了优化。今天是复盘和整合的一天。

复盘不是简单地回顾知识点，而是把散落的六天内容串成一个完整的认知体系。你需要回答一个核心问题：我能不能独立搭建一个企业知识库 Demo，从文档上传到问答输出，并且在效果不理想时知道怎么诊断和优化？

这里的实战任务是构建一个完整的企业知识库 Demo，用 5 篇文档、30 个测试问题来检验 Week 3 的阶段成果。这是一个模拟真实项目交付的场景——你有一个客户需要知识库，你需要在一天内交付一个可演示的 MVP。

核心概念

一、RAG 架构复盘

先回顾 RAG 系统的全貌。

RAG 是什么。 Retrieval-Augmented Generation，检索增强生成。核心思想是”先搜后答”——先从知识库中检索相关内容，再让大模型基于检索内容生成答案。

RAG 解决的问题。 大模型的四个核心缺陷——知识过时、缺乏私有知识、幻觉、无法引用来源——RAG 针对性地解决了这四个问题。

RAG 的两个管线。

  • 离线管线（Indexing）：文档 → 解析 → 清洗 → 切分 → 向量化 → 入库
  • 在线管线（Querying）：问题 → 向量化 → 检索 → Prompt 组装 → 生成 → 引用标注

RAG 的局限。 检索质量决定上限、长文档处理困难、多跳推理能力弱、上下文窗口有限、维护成本不低。

复盘的关键不是记住这些条目，而是能在脑中形成一张清晰的图：当你听到”RAG”这个词时，脑海里应该浮现出两条管线、六个核心模块、以及它们之间的数据流。如果浮现不出来，说明前六天的学习还不够扎实，需要回头重新梳理。

二、文档解析复盘

文档解析是 RAG 管线的第一步，也是最容易低估复杂度的环节。

核心原则。 保留有价值的内容，去除噪声，保留结构信息。

五种文档格式的解析要点：

  • PDF：最难，需要区分文字型和扫描型，注意多栏排版、表格、页眉页脚
  • Markdown：最简单，天然有结构标记
  • Word：需要从样式到结构的映射，注意表格和嵌入对象
  • Excel：不是自然语言，需要文本化或摘要化处理
  • HTML：技术最成熟，但需要处理 JS 渲染的内容

清洗的度。 宁可少去噪声，不要误杀有价值的内容。保守操作是基本原则。

元数据。 文档级别和片段级别的元数据是 RAG 系统的基础设施，从第一天就要设计好。

复盘时问自己：如果给你一份 50 页的 PDF 产品手册，你能不能设计一个解析和清洗方案，输出干净的文本 + 结构信息 + 元数据？如果不确定，回顾文档解析与清洗那一篇。

三、Chunking 复盘

切分是 RAG 系统中影响效果最显著的环节之一。

核心矛盾。 粒度要细（为了检索精准），但上下文要保留（为了理解完整）。

四种切分策略：

  • 固定长度切分：最简单，不考虑语义边界
  • 按段落切分：利用自然结构，适合大多数文档
  • 按结构切分：利用标题层级，语义完整性最好
  • 语义切分：用模型判断边界，效果最好但成本高

Overlap 的作用和代价。 避免关键信息被切断、增加检索命中率、提供上下文。但增加了存储成本和检索噪声。

Chunk Size 选择。 短 Chunk（100-300 字符）适合事实性问答，长 Chunk（800-1500 字符）适合理解性问答。大多数场景用中等 Chunk（300-800 字符）。

复盘时问自己：给定一篇技术文档，你会选择什么切分策略？为什么？如果不确定，回顾 Chunking 切分策略那一篇。

四、Embedding 和向量数据库复盘

这是 RAG 系统的”数学”部分，但工程意义大于数学细节。

Embedding 的本质。 把文本转换为向量，语义相似的文本向量距离近。这是一种有损压缩，保留核心语义，丢失细节。

两个场景。 离线生成（批量处理 Chunk）和在线生成（处理用户问题）。必须用同一个 Embedding 模型。

模型选择。 考虑维度、语言支持、性能基准（MTEB）。闭源模型质量好但付费，开源模型可以自部署。

向量数据库的作用。 向量索引加速最近邻搜索。选择依据：数据规模、部署偏好、成本预算。

相似度计算。 文本向量用余弦相似度。

复盘时问自己：我能不能解释为什么离线和在线必须用同一个 Embedding 模型？能不能在 Chroma、Pinecone、Qdrant、Weaviate 之间做出合理选择？如果不确定，回顾 前一篇 的内容。

五、RAG 链路复盘

v1 版本是”能跑通”的基线，优化版本是在基线上的改进。

六个环节。 问题处理 → 向量化 → 检索 top-k → Prompt 组装 → 答案生成 → 引用标注。

参数选择。 top-k 用 3-5，相似度阈值用 0.5，temperature 用 0-0.3。

v1 的局限。 没有查询改写、没有混合检索、没有重排序、没有上下文压缩、没有多轮对话。

六、RAG 优化复盘

优化不是一次性加入所有手段，而是诊断问题、针对性改进。

七种优化手段：

  1. Query Rewrite：改善问题表述
  2. Multi-query Retrieval：扩大检索覆盖面
  3. Hybrid Search：结合语义和精确匹配
  4. Rerank：提升排序质量
  5. Context Compression：减少 Context 长度
  6. Answer Grounding：提供细粒度引用
  7. Citation Check：验证答案准确性

优化顺序。 先测量基线，每次只加一个优化，测量改进效果，再决定下一个。

复盘时问自己：给定一个具体的 RAG 问题（比如”某些问题检索不到相关内容”），我能选择合适的优化手段吗？如果不确定，回顾 RAG 优化那一篇。

七、综合实战校准

校准题目：企业知识库 MVP

任务： 在 4 小时内构建一个企业知识库 MVP，能够回答基于 5 篇文档的问题。

文档： 自选 5 篇相关文档（或使用提供的示例文档），主题可以是 HR 政策、IT 支持、财务报销等。

要求：

  1. 离线管线（100 分）

    • 文档解析：支持 PDF 和 Markdown（30 分）
    • 文本清洗：去除噪声，保留结构信息（20 分）
    • 文本切分：选择合适的切分策略，说明理由（20 分）
    • 向量化：选择合适的 Embedding 模型，说明理由（15 分）
    • 入库：向量数据库选型，说明理由（15 分）

  2. 在线管线（100 分）

    • 问题处理：清洗，可选的分类和补全（15 分）
    • 检索：实现向量检索，可选的混合检索（25 分）
    • Prompt 组装：设计合理的 Prompt 模板（20 分）
    • 答案生成：选择合适的模型和参数（20 分）
    • 引用标注：至少实现章节级别的引用（20 分）

  3. 测试评估（100 分）

    • 设计 30 个测试问题（20 分）
    • 运行测试，记录结果（30 分）
    • 人工评分（每个问题 1-5 分）（30 分）
    • 分析得分最低的问题，找出原因（20 分）

  4. 优化改进（可选，加分项 50 分）

    • 选择 1-2 个优化手段实施
    • 重新测试，比较优化前后效果
    • 写一份优化报告

  5. 交付物（50 分）

    • 可运行的代码
    • 测试报告（包含测试问题、评分、问题分析）
    • 一页 PPT（架构图、核心设计、测试结果摘要）

评分标准：

  • 功能完整性：能否端到端运行
  • 代码质量：结构清晰、有注释、易于维护
  • 设计合理性：组件选择和参数设置有依据
  • 效果评估：测试设计合理，分析深入
  • 交付物质量：报告完整，PPT 清晰

参考时间分配：

  • 离线管线：1.5 小时
  • 在线管线：1 小时
  • 测试评估：1 小时
  • 优化改进（可选）：0.5 小时
  • 交付物整理：0.5 小时
