# SFT 与训练数据构造

如果说预训练决定了模型有没有一个大的能力底座，那么 SFT 解决的问题就是：这块底座如何被组织成“可用的行为”。

很多人第一次接触监督微调，会把它理解成“喂给模型一些问答对，让它学会回答”。这个理解不算错，但太浅了。SFT 的真正价值，不在于让模型多记住几条答案，而在于让模型把原本分散在底座里的能力，重新组织成更符合人类使用方式的接口。

说得更直接一点，SFT 不是简单地“补知识”，而是在塑造模型怎么响应指令、怎么遵守格式、怎么控制语气、怎么在一个任务边界内稳定工作。

## 一、SFT 到底在做什么

SFT 是 Supervised Fine-Tuning，监督微调。它通常建立在一个已经完成预训练的基础模型之上。训练数据不是海量原始语料，而是一批“输入-输出”样本，也就是所谓的 instruction data。

这些样本看起来很像你平常在用模型时给的 Prompt 和期望答案：

- 用户提问是什么
- 系统期望模型如何回答
- 是否要求固定格式
- 是否有风格限制
- 是否需要分步骤说明

模型通过这些样本学到的，不只是某个问题的答案，而是一种回答任务的行为模式。比如：

- 面对任务型输入时优先执行而不是续写
- 面对分类型任务时输出简洁结果
- 面对结构化任务时更稳定地遵守 JSON 或固定字段
- 面对解释型任务时使用更自然的教学式表达

这也是为什么一个 Base Model 和一个 Instruct Model 在“感觉上”差异会那么大。不是因为后者突然多了很多世界知识，而是因为它被重新教会了“如何像一个助手那样工作”。

## 二、SFT 不是万能微调

理解 SFT 时，最容易出现的误区就是把它当成通用修复工具：模型哪里不行，就拿点数据训一下。

现实里并不是这样。

SFT 最擅长解决的是以下几类问题：

- 输出格式不稳定
- 指令遵循能力不强
- 任务风格不一致
- 某类固定任务上的表现不够顺手
- 特定领域表达方式不贴近业务语境

SFT 不太擅长单独解决的问题包括：

- 基础推理能力明显不足
- 长上下文理解能力太弱
- 知识严重过时
- 根本没有接入企业私有知识
- 复杂工具使用逻辑和多阶段规划能力缺失

这些问题里，有些更适合换模型底座，有些更适合做 RAG，有些更适合改系统链路。SFT 的位置很重要，但不是万能钥匙。

## 三、高质量训练数据到底长什么样

很多人谈 SFT 时，会把注意力放在训练框架、显卡、学习率这些技术细节上。但在实际结果里，训练数据质量往往比大部分超参数都更重要。

高质量 SFT 数据通常有几个共同点。

第一，任务边界清晰。  
一条样本应该让模型很明确地知道：输入是什么，任务是什么，输出应该朝什么方向走。如果一条样本同时要求模型做分类、总结、态度判断、表格抽取，边界混乱，训练出来的行为也容易混乱。

第二，输出标准稳定。  
如果同一类任务在数据集中有三种完全不同的回答风格，模型就会学到不一致的模式。有的样本一句话回答，有的样本长篇解释，有的样本要 JSON，有的样本是自然语言，最后模型也会变得摇摆。

第三，数据代表真实使用场景。  
很多训练数据看起来格式工整，但和真实世界脱节。比如全是教科书式指令，没有口语化输入、没有残缺输入、没有模糊表达、没有边界情况。这样的模型上线后常常一碰真实用户就发散。

第四，数据覆盖关键变化。  
如果你要训练一个客服意图识别模型，不能只有标准问法。你得覆盖错别字、口语化表达、情绪化表达、上下文缺失、多意图混杂等情况。否则训练集再干净，也只是“干净地偏离现实”。

第五，尽量少噪声。  
错误标签、重复样本、相互矛盾的输出、质量参差不齐的示例，都会让模型学到混乱行为。SFT 不是数据越多越好，而是“高质量且覆盖合理”比“海量但嘈杂”更重要。

## 四、训练数据不是“收集问题答案”这么简单

真正做数据构造时，你会发现最难的不是写几条样本，而是定义数据规范。

一个成熟的数据构造过程，通常至少要明确四件事：

1. 样本结构
2. 任务分类
3. 输出标准
4. 质量检查规则

样本结构要明确到字段级别。比如一条数据是不是包含：

- system
- user
- assistant
- metadata
- task_type
- difficulty
- domain

任务分类要能告诉你这条数据属于什么问题类型，比如：

- 问答
- 抽取
- 分类
- 改写
- 总结
- 规划
- 代码

输出标准要明确“同类任务怎样算合格”。  
比如抽取任务是否必须严格 JSON，字段缺失时怎么处理；总结任务允许多长；分类任务是否必须只输出标签。

质量检查规则则决定哪些数据能进训练集。比如：

- 是否存在事实错误
- 是否输出风格和同类任务冲突
- 是否有多余解释
- 是否存在敏感信息
- 是否和其他样本高度重复

如果这四件事不清楚，后面的训练其实很容易变成“把一堆样本塞进去试试看”，最后出问题也很难定位。

## 五、训练集、验证集、测试集不是形式主义

很多人做微调时，只关心训练集，觉得“有数据能训就行”。这是一个很常见但代价很高的错误。

训练集负责让模型学习。  
验证集负责帮助你在训练过程中观察泛化趋势，判断有没有过拟合。  
测试集则是最后独立评估效果的基准，不能在训练调参过程中反复偷看。

如果你把同一批数据既拿来训练又拿来评估，得到的效果通常会比真实效果乐观得多。模型不是“真的学会了任务”，而是“学会了这批题”。

尤其在企业场景里，如果训练集和线上输入的分布差很多，表面上评测分数好看，上线之后仍然可能迅速翻车。

所以更成熟的做法是：

- 训练集覆盖主要模式
- 验证集覆盖典型输入和边界输入
- 测试集尽量模拟真实线上问题

如果资源允许，最好再保留一批真正独立的“上线前验收集”，专门防止团队在反复调参过程中把正式测试集也“看穿”了。

## 六、SFT 最怕的几类数据问题

真正做数据构造时，最值得警惕的不是“数据不够多”，而是以下几类坏数据。

第一类是标签不一致。  
同一种输入，在不同样本里对应了不同的回答风格或不同的判断逻辑。模型最怕这种冲突，因为它会学到平均化、摇摆化的行为。

第二类是伪高质量文本。  
这类数据语法工整、措辞漂亮，但不是业务真实输入。比如大量由模型自己生成的、特别“像 AI 写的”问答。它会让模型更像 AI，而不是更像能服务用户的系统。

第三类是任务泄漏。  
比如测试集中出现的问题和训练集几乎一样，只是换了个表述。这会让你高估模型的真实泛化能力。

第四类是冗余重复。  
如果某种样本在数据集中重复过多，模型就会对这类模式过拟合，而忽视其他模式。很多时候不是数据太少，而是分布太偏。

第五类是过度清洗。  
有些团队为了让数据“整齐”，会把错别字、口语、模糊表达都清掉。结果模型在训练时面对的全是理想输入，上线面对真实用户时却不适应。真实系统不是写论文，数据里必须保留现实世界的粗糙感。

## 七、SFT 的核心不是训，而是定义任务

很多训练问题最终都会回到同一个根源：任务定义不清。

如果你说“我要微调一个行业分析模型”，这句话其实没有描述任何可执行训练任务。因为“行业分析”可能包含：

- 信息抽取
- 行业概览生成
- 风险识别
- 竞争对手对比
- 结构化评分
- 报告撰写

这些任务对数据格式、输出标准、评估方式的要求都不同。如果你把它们混在一个粗糙标签下，训练数据很容易变成风格大杂烩，模型表现也会很不稳定。

更成熟的做法是先拆任务，再决定是否训练。  
比如你可以把“行业分析”拆成：

- 行业名称标准化
- 产业链结构抽取
- 竞争格局总结
- 机会点打分
- 报告生成

拆完之后再判断，哪些适合用 Prompt，哪些适合结构化输出，哪些值得做 SFT。很多时候，问题一拆开，真正需要训练的部分其实比想象中少得多。

## 八、什么时候 SFT 值得做

如果一个任务同时满足以下几个条件，SFT 往往比较值得：

- 任务重复度高
- 输出格式要求明确
- 业务风格稳定
- 有足够的高质量样本
- 单靠 Prompt 很难稳定
- RAG 也不是主要矛盾

典型例子包括：

- 特定格式的信息抽取
- 垂直领域客服回答风格统一
- 固定模板报告生成
- 特定口径的分类、打标、重写任务

如果一个任务本身变化大、知识依赖强、边界模糊、需求经常变，那 SFT 的收益通常会下降。这时更灵活的系统方式，比如 Prompt + 工具 + RAG，反而更现实。

## 九、这一篇真正要带走的东西

SFT 这件事最容易被误解成“喂数据训练”。但真正重要的不是“训”这个动作，而是三件更前置的事：

第一，你是否清楚模型到底要学什么行为。  
第二，你是否有足够稳定和真实的数据规范。  
第三，你是否知道这个问题真的值得用微调解决。

如果这三件事不清楚，训练大概率只是把混乱放大。  
如果这三件事很清楚，哪怕数据规模不算很大，SFT 也可能带来非常扎实的收益。

所以 SFT 的核心不是算力迷信，而是任务定义能力、数据构造能力和评测意识。模型训练看起来很“工程”，但最本质的部分仍然是思考清楚：你到底想让模型学成什么样。
