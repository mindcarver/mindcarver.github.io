# Day 15：RAG 原理与架构

## 学习目标

今天是 Week 3 的第一天。前两周我们学了 API 调用、Prompt 工程和结构化输出，这些能力让大模型可以按照我们的要求工作。但有一个根本性的限制始终没有解决：大模型的知识是静态的，它只知道训练数据截止日期之前的内容，对企业的私有数据一无所知。

RAG（Retrieval-Augmented Generation，检索增强生成）就是解决这个问题的核心方案。今天的目标不是写代码，而是彻底搞清楚 RAG 是什么、为什么需要它、它怎么工作、它的边界在哪里。只有把这些底层逻辑理顺了，后面六天的文档解析、切分、向量化、检索优化才有意义。

学完今天的内容，你应该能用白板给一个非技术人员讲清楚：为什么企业不能直接把文档丢给 ChatGPT，而需要一套专门的系统来做知识库问答。

---

## 核心概念

### 一、RAG 的定义

RAG 全称 Retrieval-Augmented Generation，中文翻译是"检索增强生成"。拆开来看：

- Retrieval（检索）：根据用户的问题，从外部知识库中找到相关的文档片段
- Augmented（增强）：把检索到的内容作为额外上下文，拼接到 Prompt 中
- Generation（生成）：大模型基于增强后的上下文来生成回答

用一句话概括：先搜后答。不是让大模型凭记忆回答，而是先从你的文档里找到相关内容，再让模型基于这些内容来回答。

这个思路并不新鲜。你可以把它类比为开卷考试：闭卷考试靠记忆答题（普通大模型对话），开卷考试可以翻资料答题（RAG）。没有人会否认，开卷考试在回答事实性问题时准确率高得多。

RAG 最早由 Meta（当时的 Facebook AI Research）在 2020 年的一篇论文中正式提出。原始论文探讨的是将预训练的检索器与生成模型联合训练的方法。但今天工业界说的 RAG 已经泛化了，指的是一切"先检索外部知识、再让模型生成"的模式，不一定要端到端联合训练。

### 二、RAG 解决什么问题

要理解 RAG 的价值，先得搞清楚大模型本身有哪些缺陷：

**第一，知识过时。** 大模型的知识停留在训练数据的截止日期。你问它"2024 年公司最新的退货政策是什么"，它不可能知道。即便是最强的模型，也无法回答训练之后发生的事情。

**第二，缺乏私有知识。** 大模型是在公开数据上训练的。你企业的内部文档、产品手册、客户合同、技术规范，它一概不知。你让它回答"我们公司的差旅报销标准是什么"，它会老老实实说不知道，或者更糟——编一个看起来合理的答案。

**第三，幻觉问题。** 这是大模型最令人头疼的缺陷。当它不知道答案但又想给出一个回答时，它倾向于生成一段看起来非常专业、逻辑通顺、但实际上完全错误的内容。这种"一本正经地胡说八道"在医疗、法律、金融等领域是致命的。

**第四，无法引用来源。** 即便大模型碰巧给出了正确答案，你也无法验证这个答案的出处。在企业场景中，"答案对不对"和"答案从哪里来的"同样重要。

RAG 针对的就是这四个问题。通过检索外部知识库，它让大模型可以回答训练数据之外的问题（解决过时和私有知识问题），让回答有据可依（减少幻觉），并且可以附带引用来源（可追溯）。

但要注意：RAG 不是银弹。它减少了幻觉，但没完全消灭幻觉。如果检索到的内容本身就有问题，或者模型在生成时偏离了检索到的内容，幻觉仍然会发生。这一点后面讲局限的时候会展开。

### 三、RAG 的标准流程

一个典型的 RAG 系统包含两个大阶段：离线阶段（Indexing）和在线阶段（Querying）。

**离线阶段——把文档准备好：**

1. **文档加载**。从各种来源（PDF、Word、Markdown、网页、数据库）加载原始文档。
2. **文档解析**。提取文本内容，处理不同格式的特殊问题（PDF 的表格、Word 的样式等）。
3. **文本清洗**。去除噪声（页眉页脚、多余空白、特殊字符），保留有价值的内容。
4. **结构提取**。识别标题层级、段落结构、列表、表格等，保留文档的逻辑结构。
5. **文本切分**。把长文档切成适合检索的片段（Chunk），每个 Chunk 包含完整的语义单元。
6. **向量化**。把每个 Chunk 通过 Embedding 模型转换为向量表示。
7. **入库**。把向量和原始文本存入向量数据库，建立索引。

**在线阶段——回答用户问题：**

1. **接收问题**。用户提出问题。
2. **问题向量化**。把问题通过同样的 Embedding 模型转换为向量。
3. **检索**。在向量数据库中找到与问题向量最相似的 k 个 Chunk。
4. **Prompt 组装**。把检索到的 Chunk 和用户问题组合成 Prompt。
5. **生成回答**。把组装好的 Prompt 发给大模型，生成答案。
6. **引用标注**。在答案中标注引用了哪些 Chunk、来自哪个文档。

**两个阶段的关系**。离线阶段是准备工作，做一次就行（除非文档更新）。在线阶段是实时问答，每次用户提问都会触发。离线阶段的质量直接影响在线阶段的效果——文档没解析好、Chunk 切得不合理，检索就会有问题。

### 四、RAG 的核心组件

**文档解析器**。负责从各种格式的文档中提取干净的文本。不同格式有不同的解析难点：PDF 需要处理多栏排版、扫描件需要 OCR、Word 需要处理样式和表格、Markdown 相对简单。

**文本切分器**。负责把长文档切成合适的 Chunk。切分策略有很多种：固定长度切分、按段落切分、按语义切分、递归切分。选择什么策略取决于文档类型和检索需求。

**Embedding 模型**。负责把文本转换为向量。模型的选择影响检索效果——不同模型的语义理解能力不同、生成的向量空间也不同。离线和在线必须用同一个模型。

**向量数据库**。负责存储和检索向量。主流选择包括 Chroma、Pinecone、Weaviate、Qdrant 等。不同数据库在性能、功能、部署方式上有差异。

**大模型**。负责基于检索到的上下文生成答案。可以用 OpenAI 的 GPT 系列、Anthropic 的 Claude 系列、或开源模型（Llama、Mistral 等）。

**Rerank 模型（可选）**。在向量检索之后、大模型生成之前，对检索结果做一次精排序。向量检索是粗筛，Rerank 是精筛，能提升最终检索到的 Chunk 质量。

### 五、RAG 的架构模式

**Naive RAG（朴素 RAG）**。这是最基础的架构：问题向量化 → 向量检索 top-k → 组装 Prompt → 生成答案。实现简单，适合快速验证和原型开发。

**Advanced RAG（进阶 RAG）**。在 Naive RAG 基础上增加优化模块：Query Rewrite（问题改写）、Hybrid Search（混合检索，向量+关键词）、Rerank（重排序）、Context Compression（上下文压缩）。这些优化提升了检索质量和答案准确性。

**Modular RAG（模块化 RAG）**。更灵活的架构，把 RAG 拆分为独立模块（检索器、Reranker、生成器），每个模块可以单独优化和替换。可以支持更复杂的流程（比如多轮检索、多跳推理）。

**Agentic RAG（智能体 RAG）**。把 RAG 和 Agent 结合，让系统能够自主决定是否需要检索、检索几次、用什么查询、何时停止。这是当前的前沿方向，适合复杂问答场景。

### 六、RAG 的局限

RAG 不是万能的。了解它的局限，才能避免在不适合的场景强行使用。

**检索质量决定上限。** 如果检索不到相关内容，再强的生成模型也无能为力。检索本身依赖于 Embedding 模型的语义理解能力和 Chunk 的质量。

**长文档处理困难。** 一个完整论述分散在多个 Chunk 中，检索时可能只拿到部分 Chunk，答案就不完整。这是 Chunking 的固有问题，很难完全解决。

**多跳推理能力弱。** 如果答案需要从多个不直接相关的文档中综合推理得出，RAG 的效果会明显下降。比如"根据 A 文档的规则和 B 文档的数据，计算 C"，这种任务对 RAG 很难。

**上下文窗口有限。** 虽然模型的上下文窗口越来越大（128K、200K），但把太多 Chunk 塞进 Prompt 会导致"迷失中间"现象——模型难以从大量上下文中定位到真正相关的信息。

**维护成本不低。** 文档更新需要重新解析、切分、向量化、入库。知识库规模大了之后，检索性能会下降，需要做分库、索引优化。

**不适合创意性任务。** RAG 擅长事实性问答，但不适合创意写作、头脑风暴这类需要模型发挥的任务。这类任务不需要检索，检索反而限制了模型的自由度。

## 今日总结

RAG 是"先搜后答"的问答模式，通过检索外部知识库增强大模型的生成能力。

核心价值：解决知识过时、私有知识缺失、幻觉、无法引用来源这四个问题。

标准流程分为离线阶段（文档准备）和在线阶段（实时问答）。

核心组件：文档解析器、文本切分器、Embedding 模型、向量数据库、大模型。

架构模式：Naive RAG 最简单，Advanced RAG 加入优化，Modular RAG 更灵活，Agentic RAG 最前沿。

局限：检索质量决定上限、长文档处理困难、多跳推理弱、上下文窗口有限、维护成本不低。
