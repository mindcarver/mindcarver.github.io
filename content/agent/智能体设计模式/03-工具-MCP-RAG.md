## 1. 工具使用：把能力变成受控接口

Agent 要行动，就必须通过工具接触外部世界。工具可以是搜索、数据库查询、文件读写、浏览器控制、代码执行、消息发送、部署操作，也可以是业务系统 API。

工具设计的关键不是“模型会不会调用”，而是工具是否提供了合适的边界。

一个好工具应该具备：

- 明确的参数结构。
- 清晰的成功和失败返回。
- 可审计的调用日志。
- 必要的权限和确认机制。
- 足够窄的职责。

过宽的工具会让 Agent 难以控制，例如直接给一个 `bash` 权限，模型几乎可以做任何事；过窄的工具又会让系统变成脆弱的流程脚本。实践中通常先用通用工具探索，再把高频、高风险、高价值动作提升为专用工具。

## 2. MCP：把工具接入标准化

MCP 的价值在于把模型上下文和外部能力连接方式标准化。它让 Agent 不必为每个工具单独写一套私有协议，而是通过统一接口发现工具、读取资源、调用能力。

从工程角度看，MCP 解决三类问题：

- **连接问题**：不同工具如何以统一方式暴露给 Agent。
- **上下文问题**：外部资源如何被模型按需读取。
- **治理问题**：工具能力如何被列出、授权、审计和禁用。

MCP 适合用在工具生态会持续扩张的系统里。如果只是单一应用内部的几个函数，直接函数调用可能更简单；如果要把浏览器、文档、代码仓库、数据库、云平台和内部系统都接进来，协议化就很有价值。

## 3. RAG：不是“把资料塞给模型”

RAG 的目标是让 Agent 在回答或行动前检索外部知识。它解决的是模型内置知识过期、不完整、不可追溯的问题。

但 RAG 不是简单的向量搜索。一个可用的 RAG 流程至少包含：

```text
问题理解
  -> 查询改写
  -> 多路召回
  -> 相关性排序
  -> 证据压缩
  -> 带引用生成
  -> 答案校验
```

对 Agent 来说，RAG 还要支持行动过程中的动态检索。例如代码 Agent 在修改前检索相关文件、接口文档和历史决策；客服 Agent 在回复前检索产品政策、用户订单和历史对话。

## 4. 工具、MCP、RAG 的关系

可以这样理解三者关系：

- 工具是 Agent 的行动入口。
- MCP 是工具和资源的标准接入层。
- RAG 是知识检索和证据供给机制。

一个成熟 Agent 系统往往三者都有：

```text
Agent
  -> MCP 发现可用资源和工具
  -> RAG 检索相关知识
  -> 专用工具执行受控动作
  -> 结果进入反馈循环
```

## 5. 设计建议

工具层最重要的设计原则是：把不可逆动作做窄，把只读动作做快，把高频动作做稳。

只读工具可以并行、缓存、自动执行；写入工具要有权限、确认、回滚和日志；高风险工具要尽量专用化，不要隐藏在通用 Shell 或 HTTP 请求里。

MCP 让工具生态更容易扩展，但不自动带来安全。协议只是连接层，真正的治理仍然要靠权限模型、工具描述、调用审计和人为边界。
