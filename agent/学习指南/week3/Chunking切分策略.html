Chunking 切分策略

文章导读

昨天我们把各种格式的文档解析成了干净的文本，还提取了结构信息和元数据。今天要做的是把这些长文本切成适当大小的片段——这就是 Chunking（切分）。

切分听起来简单，实际上它是 RAG 系统中影响最终效果最显著的环节之一。切得太大，检索不精准，一个 Chunk 里塞了几千字，模型很难定位到具体答案。切得太小，上下文丢失，一个只有两句话的 Chunk 缺少背景信息，模型看不懂它说的是什么。切分的粒度和方式，直接决定了检索的精度和答案的质量。

这篇文章的核心目标是理解为什么需要切分、不同的切分策略各自的优劣、Overlap 的作用和代价、以及如何评估切分质量。学完之后，你应该能够针对不同类型的文档，选择合适的切分策略，并解释为什么这样选择。

核心概念

一、为什么要切块

要理解切分的必要性，先得搞清楚为什么不能把整篇文档直接塞给模型。

向量检索需要短文本。 向量检索的原理是计算问题向量和文档片段向量之间的相似度。如果文档片段太长，向量是对整段文本的”平均表示”，会模糊掉具体细节。你问”退货的 SLA 是多少小时”，如果文档片段是一整页的内容，里面提到了退货流程、退货条件、退货SLA、退款方式等各种信息，向量表示会偏向整段文本的主题（退货相关），而不是具体的 SLA 数字。检索可能找得到这个片段，但片段太长，模型需要从中提取答案，效果打折。

上下文窗口有限。 虽然模型的上下文窗口越来越大，但把整篇 50 页的文档塞进 Prompt 是不现实的。实际操作中，检索到的片段会拼接到 Prompt 中，如果每个片段太长，能放的片段数量就少了，覆盖面变窄。

语义粒度要对齐。 用户的问题通常指向一个具体的知识点，比如一个数字、一个流程、一个定义。理想的 Chunk 大小应该和”一个完整的知识点”大致对齐。这样检索到的 Chunk 就是答案所在的最小上下文，既不缺少必要背景，又不包含太多无关信息。

存储和检索效率。 短文本的向量计算更快、存储更省、检索更高效。虽然这不是主要考虑因素，但在大规模知识库（百万级 Chunk）场景下，效率差异会很明显。

但切分也有代价。 切分打破了文档的原始结构。原本有前后逻辑关系的段落被切成了独立的片段，片段之间的连贯性丢失了。切分做得不好，可能导致一个完整的论述被腰斩，前半段在 Chunk A，后半段在 Chunk B。检索时可能只检索到 Chunk A，答案就不完整。

所以切分的核心矛盾是：粒度要细（为了检索精准），但上下文要保留（为了理解完整）。所有的切分策略都是在平衡这两个需求。

二、固定长度切分

固定长度切分是最简单的策略：按照固定的字符数或 Token 数切割文本。

怎么做。 设定一个 Chunk Size（比如 500 字符），从文本开头开始，每 500 字符切一段。最后一段如果不足 500 字符，单独作为一段保留。

优点：

  • 实现极其简单，几行逻辑就能搞定
  • 每个 Chunk 的大小一致，便于后续处理（向量维度一致、Prompt 拼接可预测）
  • 不依赖文档结构，适用于任何文本

缺点：

  • 完全不考虑语义边界，可能在句子中间、段落中间甚至单词中间切断
  • 丢失文档结构信息，标题、段落、章节的含义被忽略
  • 对于结构化文档（有明确标题层级的），效果明显不如按结构切分

适用场景： 没有结构信息的纯文本（比如 OCR 输出的无格式文本）、需要快速搭建原型的场景。

改进做法。 纯粹的固定长度切分太粗暴了。通常会做一个改进：按固定长度切，但在最近的句子边界或段落边界处切割。比如目标是 500 字符，找到 500 字符附近最近的句号或换行符，在那里切断。这样至少保证每个 Chunk 是完整的句子或段落集合。

三、按段落切分

按段落切分是利用文档的自然结构——段落作为切分边界。

怎么做。 按照换行符（单换行或双换行）识别段落，每个段落作为一个 Chunk。如果某个段落过长（超过设定的阈值），再对该段落做固定长度切分。

优点：

  • 保持了段落的完整性，语义更连贯
  • 实现简单，只需识别换行符
  • 适用于大多数自然语言文档

缺点：

  • 有些文档的段落划分不合理，可能一个段落包含多个主题
  • 段落长度差异很大，导致 Chunk 大小不均匀
  • 对于没有段落结构的文档（比如 OCR 输出），无法使用

适用场景： 有良好段落结构的文档（新闻文章、博客、报告）、中文文档（中文段落边界通常用换行符或缩进表示）。

四、按结构切分

按结构切分是利用文档的标题层级结构来切分。

怎么做。 按照标题（H1、H2、H3）来切分。比如每个 H2 下的所有内容作为一个 Chunk。如果某个 H2 下的内容太长，再按 H3 子切分。

优点：

  • 保持了文档的逻辑结构，每个 Chunk 对应一个完整的主题
  • Chunk 的语义完整性最好
  • 可以利用元数据（所属章节）来增强检索

缺点：

  • 依赖文档有良好的标题结构
  • 不同层级的标题下内容长度差异很大
  • 实现相对复杂，需要解析文档结构

适用场景： 有明确标题层级的文档（技术文档、产品手册、学术论文）、Markdown 文档。

五、语义切分

语义切分是让模型判断在哪里切分，而不是用固定规则。

怎么做。 把文本输入到一个专门训练的切分模型，模型会预测哪些地方是合适的切分边界。或者用 Embedding 模型计算相邻句子的相似度，相似度突然下降的地方可能是语义边界。

优点：

  • 真正按语义边界切分，效果最好
  • 能处理复杂的文档结构
  • 切分质量不依赖文档格式

缺点：

  • 需要额外的模型调用，成本高、速度慢
  • 实现复杂
  • 对于大文档，处理时间很长

适用场景： 对切分质量要求极高的场景、预算充足的项目、文档结构复杂的场景。

六、Overlap（重叠）

Overlap 是相邻 Chunk 之间的重叠部分。比如 Chunk 1 是 1-500 字符，Chunk 2 是 400-900 字符，重叠部分是 400-500 字符。

为什么需要 Overlap。

避免关键信息被切断。 假设一个重要论述的完整阐述是 450-550 字符，如果没有 Overlap，切分点在 500 字符处，这个论述就被切成了两半。有 Overlap 的话，两个 Chunk 都包含完整的论述。

增加检索命中率。 用户问题和 Chunk 的匹配可能在边界附近。有 Overlap 的话，即使匹配点在边界，两个 Chunk 都能检索到。

提供上下文。 每个 Chunk 有一些额外的上下文信息，模型理解 Chunk 时不会因为缺少上下文而误解。

Overap 的代价。

存储成本增加。 Overlap 意味着更多的 Chunk，更多的向量，更大的存储需求。

检索噪声增加。 Overlap 导致不同 Chunk 包含重复内容，检索时可能返回多个相似的 Chunk，浪费了检索名额（top-k）。

需要权衡。 Overlap 太小（比如 10 字符）起不到作用，Overlap 太大（比如 50%）存储和检索成本太高。常见的做法是 Chunk Size 的 10%-20%。

七、Chunk Size 的选择

Chunk Size 是切分中最关键的参数之一。太小则上下文不足，太大则检索不精准。

经验法则：

  • 短 Chunk（100-300 字符）：适合事实性问答（数字、日期、名称），但容易丢失上下文。
  • 中等 Chunk（300-800 字符）：平衡选择，适合大多数场景。
  • 长 Chunk（800-1500 字符）：适合需要上下文的理解性问答，但检索精度下降。

选择依据：

  • 问题类型。 如果用户问题通常是”XX 是多少”、“XX 怎么做”，用短 Chunk。如果用户问题需要综合理解，用长 Chunk。
  • 文档类型。 结构化文档（有章节）可以用长 Chunk，非结构化文档用短 Chunk。
  • 模型能力。 模型的上下文理解能力强，可以用长 Chunk；能力弱，用短 Chunk。
  • 实际测试。 最好的方式是实际测试：用不同 Chunk Size 在测试集上评估效果。

注意 Token 数 vs 字符数。 切分时用字符数还是 Token 数？建议用 Token 数，因为模型的上下文窗口是按 Token 计算的。中文字符和英文单词的 Token 比例不同，用字符数可能导致中英文文档的 Chunk 实际大小差异很大。

八、如何评估切分质量

切分质量直接影响 RAG 系统的效果，但如何评估呢？

定性评估：

  • 人工抽查。 随机抽取一些 Chunk，看它们是否语义完整、是否在奇怪的地方切断。
  • 案例分析。 设计一些典型问题，看检索到的 Chunk 是否包含答案。

定量评估：

  • 检索召回率。 对于已知答案的问题，答案所在的 Chunk 是否被检索到（在 top-k 中）。
  • 答案质量。 基于检索到的 Chunk 生成的答案，质量如何（可以用人工评分或自动指标）。
  • Chunk 长度分布。 统计 Chunk 长度的均值、方差、最大最小值，看分布是否合理。

迭代优化：

  1. 选择一个初始切分策略和参数
  2. 在测试集上评估效果
  3. 分析失败的案例，找出问题
  4. 调整策略或参数
  5. 重新评估，比较改进效果

核心结论

切分是 RAG 系统中影响效果最显著的环节之一。

切分的核心矛盾是粒度要细但上下文要保留。

固定长度切分最简单但不考虑语义，按段落切分利用自然结构，按结构切分保持逻辑完整，语义切分效果最好但成本高。

Overlap 能避免关键信息被切断，但增加了存储和检索成本，需要权衡。

Chunk Size 的选择依据问题类型、文档类型、模型能力，最好的方式是实际测试。

评估切分质量可以用定性评估（人工抽查、案例分析）和定量评估（检索召回率、答案质量）。
