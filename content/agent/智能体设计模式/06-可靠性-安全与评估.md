## 1. 目标设定：没有验收标准就没有可靠性

Agent 必须知道自己要达成什么，也必须知道什么叫完成。

目标应该同时包含：

- 任务目标：要交付什么。
- 质量标准：正确、完整、风格、格式、性能、安全。
- 约束条件：预算、时间、权限、不可触碰范围。
- 停止条件：何时结束，何时升级给人。

很多 Agent 失败不是因为模型能力不足，而是目标太模糊。例如“优化页面”没有边界；“把首页首屏加载时间降低 20%，不改变视觉结构，跑 Lighthouse 验证”才是可执行目标。

## 2. 异常处理：失败要可分类、可恢复

Agent 系统必须假设失败一定会发生。失败类型包括：

- 工具调用失败。
- 网络或外部服务超时。
- 权限不足。
- 输入数据缺失。
- 计划不可行。
- 输出不满足规则。
- 用户中途改变目标。

异常处理不应该只是一句“重试”。更好的方式是按失败类型决定动作：

```text
临时错误 -> 限次重试
权限错误 -> 请求授权或降级
数据缺失 -> 补充检索或声明边界
计划失败 -> 重新规划
安全风险 -> 停止并上报
验证失败 -> 回滚或修复
```

## 3. 人类参与：把人放在高杠杆节点

Human-in-the-Loop 不是让用户频繁点确认，而是在高风险、高歧义、高价值的位置引入人类判断。

适合人工确认的节点：

- 目标和范围确认。
- 不可逆操作。
- 涉及金钱、权限、合规、客户承诺的动作。
- 多个方案存在明显取舍。
- 自动验证无法覆盖的语义判断。

如果每一步都问人，Agent 失去效率；如果关键节点不问人，系统会失控。设计重点是把人工确认放在真正改变方向或风险的地方。

## 4. 护栏：提前定义不能做什么

护栏包括输入护栏、过程护栏和输出护栏。

输入护栏检查用户请求是否越权、违法、危险或缺少必要信息。过程护栏限制工具调用、权限范围、预算和访问边界。输出护栏检查敏感信息、事实引用、格式、安全建议和合规风险。

护栏不是一个单独模块，而应该贯穿 Agent 生命周期：

```text
接收任务 -> 风险分类 -> 计划审查 -> 工具门控 -> 输出校验 -> 日志审计
```

## 5. 评估与监控：上线后才是真测试

Agent 的评估不能只看单次输出好不好。需要同时看任务成功率、工具失败率、人工介入率、平均成本、延迟、回滚次数、安全事件、用户修正次数。

离线评估用于上线前筛选能力，在线监控用于发现真实环境中的漂移。

建议至少建立三类评估集：

- 标准任务集：覆盖主要成功路径。
- 对抗任务集：覆盖越权、误导、提示注入、异常输入。
- 回归任务集：覆盖过去失败过的真实案例。

## 6. 我的理解

Demo 展示的是 Agent 能做什么；系统设计关注的是 Agent 什么时候不能做、做错了怎么办、谁来发现、如何恢复。

可靠性不是靠提示词承诺出来的，而是靠目标、工具边界、异常处理、人工关口、护栏和监控共同构建的。
