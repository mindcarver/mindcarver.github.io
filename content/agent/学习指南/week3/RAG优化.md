# RAG 优化

## 文章导读

昨天我们跑通了 RAG 问答链路 v1，一个端到端的系统已经能工作了。但你一定发现了不少问题：有些问题检索不到相关内容、有些检索到了但答案质量不高、有些问题的答案引用了错误的来源。

今天是"找问题、改问题"的一天。我们要学习 RAG 系统中最常见的优化手段：Query Rewrite、Multi-query Retrieval、Hybrid Search、Rerank、Context Compression、Answer Grounding、Citation Check，以及检索失败时的回退策略。

这些优化手段不是都要加上的。正确的做法是：先用 v1 跑出基线数据，找出最影响效果的一两个问题，针对性地优化。每次只改一个变量，测量改进效果，然后再改下一个。

学完今天，你应该能诊断 RAG 系统中的常见问题，选择合适的优化手段，并且能量化优化前后的效果差异。

---

## 核心概念

### 一、Query Rewrite（查询改写）

Query Rewrite 是在用户原始问题进入检索之前，对问题进行改写，使其更适合检索。

**为什么需要改写。** 用户的问题表述方式和文档中的表述方式经常不一致。用户可能问"东西坏了怎么退"，但文档中写的是"产品退换货流程"。两段文字语义相近，但字面重叠很少，纯向量检索可能匹配不上。

用户的问题还可能太简短（"退货？"）或者太模糊（"那个政策怎么说"），缺乏足够的信息来生成有效的检索向量。

**改写的方式：**

**扩展式改写。** 将简短的问题扩展成更完整的表述。比如"退货？"扩展为"产品的退货流程和退货条件是什么"。

**关键词提取式改写。** 从问题中提取核心关键词，用于关键词检索。比如"购买超过 30 天还能退货吗"提取出"购买超过30天"、"退货"等关键词。

**同义替换式改写。** 将用户用词替换为文档中可能使用的术语。比如"东西坏了"替换为"产品质量问题"或"产品缺陷"。

**用大模型做改写。** 最灵活的方式是让大模型来改写查询。给模型一个 Prompt："请将以下用户问题改写为更适合在知识库中检索的查询语句，保留核心意图，使用更正式和完整的表述。"模型通常能生成比原始问题更适合检索的查询。

**改写的风险。** 改写可能改变用户的原始意图。如果改写后的查询偏离了用户想问的内容，检索结果就会偏。所以改写后需要验证：检索到的 Chunk 是否确实和用户的原始问题相关，而不是只和改写后的查询相关。

### 二、Multi-query Retrieval（多查询检索）

Multi-query Retrieval 的思路是：与其用一个问题检索一次，不如生成多个不同角度的查询，分别检索，然后合并结果。

**为什么需要多查询。** 用户的问题只代表了一种表述方式，但同一个意图可以用很多不同的方式表达。一个查询可能检索不到某些相关内容，但另一个查询可以。多查询增加了"命中"的概率。

**怎么做：**

1. 用大模型将用户问题改写为 3-5 个不同角度的查询
2. 对每个查询分别做向量检索（每个查询返回 top-k）
3. 将所有检索结果合并，去重
4. 按相似度重新排序，取最终的 top-k

**举个例子。** 用户问"退货要多久"：

- 查询1："退货的时限是多少天"
- 查询2："退货处理时间"
- 查询3："退款到账时间"

三个查询可能检索到不同侧面的 Chunk，合并后覆盖面更广。

**代价。** 多查询意味着更多的 Embedding API 调用和更多的向量检索操作，延迟和成本都会增加。需要在效果和性能之间权衡。

### 三、Hybrid Search（混合检索）

向量检索擅长语义匹配，但不擅长精确匹配。比如"SKU12345 这个产品多少钱"，向量检索可能检索不到，因为 SKU 编号是精确匹配的问题。关键词检索（BM25）擅长精确匹配，但不擅长语义理解。Hybrid Search 把两者结合起来。

**怎么做：**

1. 对用户问题分别做向量检索和关键词检索
2. 两种检索各返回 top-k 结果
3. 用加权算法合并两种结果（比如向量相似度权重 0.7，关键词相关性权重 0.3）
4. 按合并后的分数重新排序，取最终的 top-k

**什么时候需要 Hybrid Search。** 当你的知识库包含很多专有名词、产品编号、技术规格等需要精确匹配的内容时，Hybrid Search 能明显提升效果。如果知识库主要是自然语言描述，纯向量检索通常就够了。

### 四、Rerank（重排序）

Rerank 是在向量检索之后，用更精准的模型对检索结果做一次精排序。

**为什么需要 Rerank。** 向量检索是粗筛，速度快但不够精准。它返回的 top-k 结果中，可能有一些并不真正相关。Rerank 模型会仔细计算问题和每个候选 Chunk 的相关性，重新排序，把最相关的排在前面。

**Rerank 模型。** 专门训练的交叉编码器（Cross-Encoder）模型，如 BGE-Reranker、Cohere Rerank。这类模型同时输入问题和候选 Chunk，输出一个相关性分数。

**怎么做：**

1. 向量检索返回 top-20 或 top-50 结果（比平时多）
2. 用 Rerank 模型对这 20-50 个结果逐一打分
3. 按分数重新排序，取最终的 top-3 或 top-5

**代价。** Rerank 模型的计算成本比向量检索高得多，因为它需要对每个候选 Chunk 做一次推理。所以通常先用向量检索做粗筛，减少候选数量，再用 Rerank 做精排。

### 五、Context Compression（上下文压缩）

从向量数据库检索到的 Chunk 可能很长，包含很多无关信息。如果把这些 Chunk 直接塞进 Prompt，会浪费上下文窗口，还可能干扰模型理解。Context Compression 的目标是从每个 Chunk 中提取最相关的部分。

**怎么做：**

一种方法是用 LLM 提取："请从以下文本中提取与问题最相关的句子，去除无关内容。"

另一种方法是用专门的压缩模型，如 LLMLingua-2，可以在保留关键信息的同时大幅压缩文本长度。

**效果。** 压缩后的 Context 更短、更精准，模型更容易从中找到答案。但压缩本身也调用 LLM，增加了成本和延迟。

### 六、Answer Grounding（答案定位）

Answer Grounding 是在生成答案的同时，标注答案的每个部分来自哪个 Chunk 的哪一部分。

**为什么需要。** 引用标注让用户可以追溯答案来源，但粗粒度的引用（"来自文档 X"）可能不够。细粒度的 Grounding（"答案第 2 句来自文档 X 第 3 段第 2 句"）让用户可以验证答案的准确性。

**怎么做。**

一种方法是在 Prompt 中要求模型："请回答问题，并在每个句子后面用括号标注来源，如（来源1 第3段）。"

另一种方法是用专门的 Grounding 模型，它会在生成答案的同时输出引用的位置信息。

### 七、Citation Check（引用验证）

Citation Check 是验证模型生成的答案是否真的来自检索到的内容，还是模型在编造。

**怎么做。**

一种方法是用规则检查：答案中的关键实体、数字、结论，是否能在检索到的 Chunk 中找到对应内容。

另一种方法是用 LLM 验证："请检查以下答案是否都能从给定的知识库内容中推导出来。如果有不能推导的部分，请指出。"

如果验证失败，可以：
- 返回一个保守的答案："根据知识库，X 是明确的，但 Y 需要进一步确认。"
- 要求模型重新生成，要求更严格地基于检索内容

### 八、检索失败的回退策略

当检索不到相关内容时（所有 Chunk 相似度都低于阈值），怎么处理？

**策略1：直接告诉用户未找到。** 最诚实的方式："抱歉，知识库中没有找到相关信息。"

**策略2：使用模型的通用知识。** 告诉模型："知识库中没有找到相关信息。请基于你的通用知识回答，但要明确说明这是通用知识，不是来自知识库。"

**策略3：放宽检索条件。** 降低相似度阈值，扩大检索范围，看是否有勉强相关的内容。

**策略4：引导用户重新表述。** "没有找到相关信息。您是不是想问 XXX？或者能否提供更多细节？"

**选择建议。** 企业场景建议用策略1或策略4。不准确但有帮助的答案不如诚实的"不知道"。

### 九、优化顺序建议

不要一次性加入所有优化。按照以下顺序逐步优化，每一步测量效果：

1. **基线测量。** 用 v1 版本在测试集上跑一遍，记录准确率、召回率等指标。
2. **Query Rewrite。** 如果用户问题表述不精确是主要问题，先加这个。
3. **Rerank。** 如果检索结果排序有问题，加这个通常收益最大。
4. **Hybrid Search。** 如果有很多精确匹配需求，加这个。
5. **Multi-query。** 如果经常检索不到相关内容，考虑这个。
6. **Context Compression。** 如果 Context 太长影响效果，加这个。
7. **Answer Grounding / Citation Check。** 最后加上，提升可信度。

每次只加一个优化，测量改进效果。如果改进不明显，可能这个优化不适合你的场景，或者有其他更重要的问题。

## 核心结论

RAG 优化有很多手段，但不要一次性全部加上。

Query Rewrite 改善问题表述，Multi-query 扩大检索覆盖面。

Hybrid Search 结合语义和精确匹配，Rerank 提升排序质量。

Context Compression 减少 Context 长度，Answer Grounding 提供细粒度引用。

Citation Check 验证答案准确性，检索失败时有多种回退策略。

优化顺序：先测量基线，每次只加一个优化，测量改进效果，再决定下一个。

## Week 3 综合练习

### 练习：企业知识库 Demo

用 5 篇文档构建一个完整的企业知识库 Demo：

**文档选择：** 找 5 篇相关文档（比如一个公司的 HR 政策、IT 支持、财务报销、产品手册、销售流程各一篇）。

**数据处理：** 解析文档、清洗、切分、向量化、入库。

**测试问题：** 设计 30 个测试问题，覆盖：
- 10 个简单事实问题（"XX 是多少"、"XX 怎么做"）
- 10 个中等复杂度问题（需要综合 2-3 个 Chunk）
- 10 个困难问题（跨文档推理、需要理解上下文）

**评估：** 运行 v1 版本，记录每个问题的答案质量（人工评分 1-5 分）。找出得分最低的问题，分析原因，针对性优化。

**优化：** 选择 1-2 个优化手段，重新评估，比较改进效果。

**交付：** 一个可演示的 RAG 系统、一份测试报告（包含测试问题、评分、优化前后对比）。
