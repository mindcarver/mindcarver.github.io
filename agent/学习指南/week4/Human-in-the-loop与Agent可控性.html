Human-in-the-loop 与 Agent 可控性

文章导读

前文设计了一个 6 Agent 的行业研究系统，它能自动完成从行业分析到方案生成的全流程。但有一个问题悬而未决：如果 Agent 做出了错误的判断，怎么办？如果它推荐了一个实际上不可行的方案，报告已经发出去了，怎么挽回？

这就是 Human-in-the-loop 要解决的核心问题。不管 Agent 有多智能，在当前技术水平下，完全自主的 Agent 是不可靠的。我们需要在关键节点加入人工审核和干预机制，让人类能在 Agent 执行过程中进行检查、修正和批准。

这篇文章的核心目标是建立系统化的人工介入设计能力。读完这一部分，你应该能识别哪些节点需要人工审核、设计审核表单和确认机制、理解中断与继续的实现逻辑、建立可追踪的执行记录体系。

核心概念

为什么 Agent 需要人工审核

先回答一个根本性的问题：既然 Agent 这么智能，为什么还需要人工介入？

原因有三个，每一个都是当前技术的硬约束。

第一个原因是大模型的输出不完全可信。模型会产生幻觉，会生成看似合理但实际错误的内容。在行业分析场景中，模型可能虚构不存在的市场数据、错误描述某个行业的产业链结构、把过时的信息当成最新数据。这些问题如果不经过人工审核就输出给客户，后果可能很严重——轻则丢面子，重则丢客户。

第二个原因是决策中存在主观偏好。AI 方案的推荐不只是技术判断，还涉及客户的业务策略、风险偏好、预算约束、组织文化。模型不了解这些上下文，可能推荐一个技术上最优但客户组织无法落地的方案。人工审核的价值在于注入模型缺少的业务上下文和主观判断。

第三个原因是责任归属。企业级的 AI 应用必须有明确的责任链条。如果一份报告给了客户错误的建议，谁负责？如果全程自动、无人审核，责任无法追溯。加入人工审核环节，审核人就是该节点的责任人。这不仅是一个管理要求，也是企业客户的基本诉求。

从更宏观的角度看，Human-in-the-loop 不是一个临时补丁，而是 AI 系统设计的基本原则。在可预见的未来，AI 的能力边界决定了它在企业应用中必须有人类监督。这不是”不信任 AI”，而是”合理使用 AI”。

哪些节点需要人工审核

不是所有节点都需要人工审核。过度的人工介入会抵消自动化的效率优势。关键是识别出”必须由人来把关”的节点。

判断标准有三条。第一条，这个节点的输出如果出错，后果有多严重？严重就加审核。第二条，这个节点的判断是否涉及主观偏好或业务上下文？涉及就加审核。第三条，这个节点的输出是否会影响外部系统或人员？会影响就加审核。

拿行业研究系统的 6 个 Agent 来看：

Industry Research Agent 的输出（行业概览）：建议加审核。行业概览是后续所有分析的基础，如果底层数据错了，后面的分析全部建立在错误前提上。审核重点是数据的准确性和概览的完整性。

Role Analyst Agent 和 Process Analyst Agent 的输出：可以不加审核（除非客户特别要求）。这两个 Agent 的分析结果相对客观，而且后面还有 AI Solution Agent 和 Risk Review Agent 会交叉验证。

AI Solution Agent 的输出（AI 方案推荐）：建议加审核。方案推荐直接影响客户的投资决策，涉及技术可行性、成本估算、实施周期等判断，出错后果严重。

Risk Review Agent 的输出（风险审查）：必须加审核。风险审查是对整个方案的最终把关，审核人通常是项目负责人或领域专家。

最终报告输出：必须加审核。这是交付给客户的东西，必须经过人工确认。

总结一下，行业研究系统建议在四个节点加人工审核：行业概览生成后、AI 方案推荐后、风险审查后、最终报告生成后。从效率角度看，这意味着整个流程有四次暂停等待人工确认。如果觉得太频繁，可以合并后两个审核点：风险审查和最终报告一起审核。

审核表单设计

人工审核不是让人看一眼原始输出就行。要设计清晰的审核表单，让审核者快速抓住重点、做出判断。

审核表单包含三个部分：关键信息摘要、审核选项、审核意见。

关键信息摘要是 Agent 输出的精简呈现。不要让审核者看 5000 字的原始输出，而是提炼出最关键的信息点。比如行业概览的审核表单，关键信息包括：行业名称、市场规模、产业链结构、主要玩家、核心技术。每个信息点用一两句话呈现，附带数据来源。

审核选项是给审核者的选择。通常包括：通过、拒绝、要求修改。如果选”要求修改”，需要填写修改意见。

审核意见是审核者的自由文本输入，用于记录通过的原因、拒绝的理由、修改的具体要求。

审核表单要简洁。一个原则是”一屏看完”。如果审核表单需要滚动好几屏，审核者会失去耐心。

中断与继续

人工审核会中断 Agent 的执行流程。需要设计清晰的”中断→审核→继续”机制。

中断时，Agent 暂停执行，保存当前状态，向审核者发送通知。通知包含：任务名称、当前节点、待审核内容、审核链接。

审核者审核后，有三个可能的结果：通过、拒绝、要求修改。

如果通过，Agent 从中断点继续执行。继续时要考虑一个问题：审核过程中可能经过了较长时间（比如几个小时后审核者才看），环境可能已经变化。比如外部 API 的数据可能更新了、缓存可能失效了。继续前要判断是否需要刷新某些数据。

如果拒绝，Agent 终止执行，返回拒绝原因。可能需要回滚已执行的部分操作（如果涉及外部系统写入）。

如果要求修改，Agent 根据修改意见重新执行该节点。重新执行可能需要重新调用模型、重新获取数据、重新生成输出。

可追踪的执行记录

人工介入意味着责任链条。必须有完整的执行记录，记录每个节点做了什么、谁审核的、审核结果是什么、什么时候审核的。

执行记录包含：任务 ID、节点名称、执行时间、Agent 输出、审核人、审核时间、审核结果、审核意见。这些记录不仅是责任追溯的依据，也是系统优化的重要数据——通过分析哪些节点经常被拒绝、哪些修改意见最常见，可以改进 Agent 的设计和 Prompt。

执行记录要安全存储、防止篡改、控制访问权限。只有授权人员才能查看审核历史。

Human-in-the-loop 的成本

人工审核不是没有代价的。主要成本有两类。

第一类是时间成本。审核需要时间，审核周期可能是几分钟到几小时不等。这意味着整个 Agent 流程的执行时间会大幅增加。如果追求快速响应，可能需要 24/7 的审核团队，这又增加了人力成本。

第二类是人力成本。审核需要人，而且是懂业务的人。行业研究需要行业专家，技术方案需要技术专家。这些专家的时间是昂贵的。

平衡人工审核和自动化效率是设计的关键。实践中可以采用分层审核策略：高风险节点必须人工审核，中风险节点抽样审核，低风险节点自动运行但定期审计。

可控性设计

Human-in-the-loop 是让 Agent 可控的最后一道防线。但可控性应该从更早的设计阶段就开始考虑。

可控性设计的五个原则：

第一，目标边界清晰。Agent 的目标要明确、可验证、有边界。模糊的目标（“分析一下这个行业”）会让 Agent 的行为不可预测。

第二，工具权限控制。如前文所讲，工具白名单、参数限制、操作审批三层防护。

第三，步数限制。限制 Agent 的最大执行步数，防止无限循环。比如设置最多 20 步，超过后强制终止。

第四，成本上限。设置 Token 消耗和 API 调用的上限，超过后暂停执行，发出告警。

第五，人工介入权。在关键节点预留人工介入接口，允许随时中断、审核、终止。

这些原则和 Human-in-the-loop 一起，构成了 Agent 系统的可控性保障。

核心结论

人工审核的三个原因：模型输出不完全可信、决策涉及主观偏好、责任归属需要明确。

判断审核节点的三条标准：后果严重性、主观偏好、外部影响。

审核表单包含：关键信息摘要、审核选项、审核意见。要简洁，一屏看完。

中断与继续机制：通过、拒绝、要求修改三种结果。

执行记录要完整，用于责任追溯和系统优化。

可控性设计五个原则：目标边界清晰、工具权限控制、步数限制、成本上限、人工介入权。
