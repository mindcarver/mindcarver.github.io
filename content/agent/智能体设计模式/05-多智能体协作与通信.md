# 多智能体协作与通信：拆分角色之前先拆分责任

## 1. 多智能体不是越多越好

多智能体协作的直觉很诱人：把复杂任务拆给多个专家，让它们并行工作。但智能体数量增加后，系统复杂度不是线性增加，而是快速上升。

多智能体真正适合的场景是：

- 子任务相互独立，可以并行推进。
- 需要不同专业视角，例如安全、性能、产品、实现。
- 需要生成与审查分离，降低自证偏差。
- 单个上下文窗口无法容纳全部信息。
- 不同角色需要不同工具权限。

不适合的场景：

- 任务很小，协调成本高于执行成本。
- 子任务强耦合，需要频繁同步。
- 没有清晰的最终仲裁者。
- 多个智能体会同时写同一片文件或数据。

## 2. 角色拆分要基于责任，不是头衔

常见错误是先起角色名：架构师、开发、测试、审查员。更好的方式是先问责任边界：

- 谁负责定义目标？
- 谁负责收集证据？
- 谁负责实现？
- 谁负责验证？
- 谁负责质疑？
- 谁拥有最终决策权？

角色只有在责任不同、工具不同、上下文不同或判断标准不同时才有意义。否则多个 Agent 只是重复工作。

## 3. 通信协议比智能体本身更重要

多智能体系统的失败常常不是某个 Agent 不聪明，而是通信没有约束。

通信需要明确：

- 消息格式：任务、输入、输出、证据、风险、状态。
- 生命周期：待办、进行中、阻塞、完成、失败。
- 冲突处理：结果不一致时谁仲裁。
- 共享状态：哪些信息写入公共空间，哪些只在局部保留。
- 权限边界：谁能读、谁能写、谁能执行高风险操作。

A2A 的价值在于把智能体间交互协议化。协议化之后，协作不再依赖“自然语言互相喊话”，而是有结构化状态、能力声明和任务交接。

## 4. 三种常见协作结构

### 4.1 主从结构

一个主 Agent 负责规划、分派和整合，多个子 Agent 执行独立任务。这是最容易落地的结构。

适合代码库分析、资料调研、多模块修复、并行审查。

### 4.2 评审结构

一个 Agent 生成方案，另一个或多个 Agent 专门审查。评审者不参与实现，减少立场污染。

适合架构方案、安全审查、代码 review、重要文档定稿。

### 4.3 市场结构

多个 Agent 提出候选方案，系统按评分、成本、风险或用户偏好选择。这种结构探索能力强，但协调成本高。

适合创意生成、策略设计、开放式研究。

## 5. 我的实践判断

多智能体设计的第一原则是：能不拆就不拆，必须拆时先定义所有权。

尤其是代码任务，最重要的是写入边界。一个 Agent 负责认证模块，另一个负责数据库迁移，第三个负责测试，这样可以并行；如果三个 Agent 都改同一个核心文件，就会制造冲突。

多智能体不是为了“看起来高级”，而是为了把不同认知任务隔离：探索、实现、验证、批判、总结。隔离之后，系统才更可靠。
