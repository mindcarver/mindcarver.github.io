## 1. 提示链：把复杂任务拆成可检查步骤

提示链的核心思想是：不要让模型一次性完成所有推理，而是把任务拆成多个中间步骤，每一步产生一个可检查产物。

例如写一篇技术文章，可以拆成：

```text
资料提取 -> 主题归纳 -> 大纲生成 -> 分节写作 -> 事实校验 -> 风格润色
```

提示链的价值不只是提升质量，更重要的是让失败位置可定位。一次性生成失败时，我们不知道是资料理解错了、结构错了，还是表达错了；链式流程可以把错误锁定到具体节点。

适用场景：

- 输出质量要求高。
- 中间推理必须可审查。
- 任务可以自然拆成阶段。

不适用场景：

- 简单问答。
- 低风险、低成本的一次性生成。
- 强实时交互，无法承受多轮延迟。

## 2. 路由：先判断任务类型，再选择路径

路由模式解决的是“所有请求都走同一个处理链”的低效问题。

一个 Agent 系统通常会面对多种输入：简单问答、复杂分析、代码修改、资料检索、工具操作、人工审批。不同输入需要不同模型、工具、预算和安全策略。

路由层可以按这些维度分流：

- 任务意图：查询、写作、修改、执行、审查。
- 风险等级：只读、可逆写入、不可逆操作。
- 复杂度：单步、多步、需要规划、需要多智能体。
- 数据来源：本地仓库、网页、数据库、内部文档。
- 输出形态：答案、文件、代码、报告、操作结果。

路由的难点是误分流。误把高风险任务当低风险任务，比慢一点更危险。所以路由结果应该带置信度，低置信度时进入保守路径。

## 3. 并行化：把独立子任务同时推进

并行化适合处理互不依赖的任务，例如：

- 同时检索多个资料来源。
- 同时让多个审查角色检查同一份方案。
- 同时运行测试、类型检查和 lint。
- 同时让不同智能体分析不同模块。

并行化的收益是吞吐量，代价是协调复杂度。真正的问题不是“能不能并发”，而是“结果怎么合并”。

一个可控的并行流程需要三件事：

- 每个子任务边界清楚。
- 子任务之间写入范围不冲突。
- 汇总层有明确的冲突处理规则。

如果多个智能体同时编辑同一文件，但没有所有权边界，就会把速度优势变成合并灾难。

## 4. 规划：让 Agent 先建立任务地图

规划模式用于处理无法一步完成的任务。计划不是为了形式感，而是为了提前暴露依赖、风险和验收标准。

好的计划应该包含：

- 目标：最终要交付什么。
- 范围：哪些内容做，哪些不做。
- 步骤：先后顺序和依赖关系。
- 验证：每一步如何证明完成。
- 风险：可能失败的点和回退路径。

规划也不能过度。低风险小任务先写长计划，会让 Agent 变慢。我的经验是：只要任务跨多个文件、多个系统、多个阶段，或者失败成本明显，就应该先规划；否则直接执行再轻量验证。

## 5. 组合方式

这四种模式经常组合使用：

```text
路由判断任务类型
  -> 复杂任务进入规划
  -> 规划拆成提示链
  -> 独立步骤并行执行
  -> 汇总结果并验证
```

基础工作流模式的目标是把 Agent 从“自由发挥”变成“有结构地行动”。结构不是限制智能，而是降低失控概率。
