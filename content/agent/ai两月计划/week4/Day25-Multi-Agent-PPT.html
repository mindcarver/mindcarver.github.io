25 / MULTI-AGENT ARCHITECTURE

从单 Agent 到协作系统

Multi-Agent 的核心不是“更多模型”，而是把复杂任务拆成多个职责明确、工具精简、可协调的专家单元，用工程化方式提升决策质量与可控性。

Theme: Multi-Agent Architecture Pattern: Orchestrator + Specialists Scope: 6-Agent Research Team
Focus: value, decomposition, communication, conflict resolution, integration
01 / WHY NOW

单 Agent 的三个瓶颈，决定了何时需要升级架构

Multi-Agent 不是炫技选项，而是当任务复杂度、工具数量、上下文长度同时上升时的一种必要工程手段。

原则先行：能用单 Agent 解决的，不要用 Multi-Agent。只有在任务存在明确专业分工边界时，引入多智能体才有价值。

01
TASK COMPLEXITY

行业分析、岗位拆解、流程优化、方案设计、风险审查无法由一个 Agent 同时深挖。

02
TOOL OVERLOAD

工具越多，选择压力越大，误选概率上升；拆分后每个 Agent 仅需 2–3 个核心工具。

03
CONTEXT DRIFT

长历史对话会稀释模型注意力；局部职责让上下文更短、更聚焦。

02 / TRADE-OFFS

收益不是免费的：复杂度、通信、成本同步上升

多 Agent 提升专业化，也引入新的系统负担。架构收益必须大于协调成本。

关键判断：如果协作、审计、重试、监控带来的系统负担超过任务本身收益，架构就被设计过度了。

6×
AGENT NODES

从 1 个节点扩展到 6 个节点，调试面、故障面与依赖面显著放大。

+1
NEW FAILURE LAYER

通信与协调成为新的故障点，信息可能丢失、变形或延迟。

3–4×
TOKEN COST

总 Token 通常高于单 Agent，虽然不会线性放大到 6 倍。

03 / CONTROL PLANE

Orchestrator 是管理者，不是专家

主控 Agent 的价值在于拆任务、排依赖、收结果、控异常。它不做专业分析，否则会重新滑回“全能单 Agent”的陷阱。

Role: Orchestrator Function: Coordination Anti-pattern: Do-analysis-itself
Design rule: separate control plane from expertise plane
04 / ORCHESTRATOR JOBS

四项核心职责，定义主控 Agent 的边界

任务分解

将用户请求拆为可执行子任务，并映射到对应专家 Agent。

执行调度

管理并行与串行关系，确定依赖顺序与触发条件。

结果整合

回收输出，检查质量与完整性，组织成最终报告。

异常处理

面对失败、超时、低质量输出，执行重试、降级、跳过或报错策略。

设计戒律：Orchestrator 不应该拥有自己的行业研究、岗位分析或方案设计能力。它只负责流程，不负责内容专业判断。

05 / SPECIALIST DESIGN

专家 Agent 的质量，取决于边界是否足够清晰

好的专家 Agent 不是“更聪明”，而是“更专注”。职责边界、工具集、Prompt 三者必须同时收缩。

Specialist pattern: narrow scope, fewer tools, stable prompt
1
SINGLE RESPONSIBILITY

一个 Agent 只做一类专业任务，避免职责重叠与冲突。

2–3
CORE TOOLS

工具精简降低选择负担，提高调用准确率与稳定性。

FOCUSED
SYSTEM PROMPT

Prompt 仅覆盖本领域目标、方法与输出格式，不承载全局逻辑。

06 / TEAM TOPOLOGY

一个 6-Agent 行业研究团队的典型编制

ORCH
ORCHESTRATOR

任务分解、调度、整合、异常处理。

IND
INDUSTRY RESEARCH

产业链、市场规模、主要玩家、发展趋势。

ROLE
ROLE ANALYST

岗位列表、职责描述、技能要求。

PROC
PROCESS ANALYST

流程梳理、瓶颈识别、优化建议。

AI
AI SOLUTION

场景清单、技术方案、优先级评分。

RISK
RISK REVIEW

风险清单、合规问题、应对建议。

This layout separates expertise by domain instead of by model capability.

07 / EXECUTION GRAPH

并行先收集，串行再推导

Industry Research、Role Analyst、Process Analyst 可并行启动；AI Solution 依赖前三者；Risk Review 依赖方案输出。调度图决定吞吐与可靠性。

Mode: Parallel + Sequential Dependency: Explicit Output: Structured Report
A good orchestration graph reduces idle waiting and redundant calls.
08 / PIPELINE

建议执行流：先分析环境，再生成方案，最后审查风险

Step 1

Orchestrator 接收请求，解析目标与交付物。

Step 2

Industry / Role / Process 三个 Agent 并行执行基础分析。

Step 3

AI Solution Agent 基于三份分析结果生成应用方案。

Step 4

Risk Review Agent 审查方案可行性、风险与限制。

Step 5

Orchestrator 合并、校验、修订并输出最终报告。

调度原则：先收集事实与结构，再做推理与设计，最后做审查与否决。

09 / COMMUNICATION MODES

通信不是细节，而是系统可靠性的主战场

Agent 间如何交换信息，决定了透明度、效率与可监控性之间的平衡。

INDIRECT
VIA ORCHESTRATOR

优点是全局可见、易监控；缺点是路径更长，信息可能延迟或损耗。

DIRECT
PEER TO PEER

优点是高效灵活；缺点是主控失去完整信息流视图。

HYBRID
PRACTICAL DEFAULT

关键信息由主控统一分发，局部临时确认允许直接对话。

10 / MESSAGE PROTOCOL

结构化 JSON 协议，让消息可解析、可追踪、可审计

自由文本适合表达，不适合系统协作。协议化消息是 Multi-Agent 工程化落地的基础设施。

Recommended fields: message_type, sender, receiver, content, timestamp.

TYPE
MESSAGE TYPE

区分 request、result、clarification、error、retry 等消息类别。

ROUTE
SENDER / RECEIVER

明确消息流向，便于回放链路与定位责任节点。

TRACE
TIME / CONTENT

保留时间戳与结构化正文，支撑监控、审计与故障排查。

11 / CONFLICT MANAGEMENT

多专家输出不一致，是常态，不是例外

当不同 Agent 给出不同数据、不同结论、不同建议时，系统必须有一套明确定义的冲突处理机制，而不是临时拍脑袋。

Conflict: Data Conflict: Judgment Conflict: Priority
Unresolved conflicts silently corrupt downstream decisions.
12 / RESOLUTION STRATEGIES

三种策略解决冲突：规则、人工、协商

优先级规则

预先定义谁拥有最终裁决权，例如 Risk Review 的否决优先级最高。

人工仲裁

对高风险、低置信度、无法自动判定的冲突暂停流程并提交人工审核。

协商机制

相关 Agent 交换证据、来源、时间新旧与可靠性，收敛到一致结论。

实践建议：先用规则解决大部分冲突，把人工仲裁留给高价值或高风险节点，再把协商机制作为中间层补足自动化弹性。

13 / RESULT INTEGRATION

整合不是拼接，而是一次最终质量控制

Orchestrator 在最后阶段要像编辑部总编一样工作：补全、对齐、统一、定稿。

CHECK
COMPLETENESS

确认所有必须提交的 Agent 已返回结果，格式满足要求。

RESOLVE
CONFLICTS

对冲突数据、矛盾建议、风险争议执行决策与修订。

COMPOSE
COHERENT OUTPUT

生成术语一致、章节清晰、逻辑连贯的最终结构化报告。

14 / DESIGN TAKEAWAYS

架构原则总结：少即是多，边界决定质量

一个可用的 Multi-Agent 系统，关键不在 Agent 数量，而在是否具备清晰分工、可观测通信、可执行调度与可落地的冲突治理。

Next step:当 Agent 判断错误时，如何引入人工审核与人工干预？这正是 Human-in-the-loop 要解决的问题。

RULE 1
USE WHEN NEEDED

只有任务跨多个专业域且边界明确时，才值得上 Multi-Agent。

RULE 2
KEEP AGENTS NARROW

职责越窄，工具越少，Prompt 越聚焦，结果越稳定。

RULE 3
MAKE FLOW OBSERVABLE

通信协议、调度图、冲突日志与重试策略必须可追踪。
