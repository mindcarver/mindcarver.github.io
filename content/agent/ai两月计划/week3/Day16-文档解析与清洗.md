# Day 16：文档解析与清洗

## 学习目标

昨天我们建立了 RAG 系统的全景认知，知道了"先搜后答"的完整流程。今天开始进入第一个工程环节：文档解析与清洗。

这是整个 RAG 管线的起点。如果这一步做不好，后面的切分会乱、向量化会差、检索会偏、答案自然好不了。可以把它比作做菜之前的食材处理——鱼鳞没刮干净，调料再好也白搭。

今天的目标是理解企业中常见的文档格式、每种格式的解析难点、清洗的原则和方法、以及元数据提取的标准。学完之后，你应该能够为一个企业知识库项目设计出合理的文档处理方案，知道面对一堆乱七八糟的文档该从哪里下手。

---

## 核心概念

### 一、PDF 解析

PDF 是企业中最常见的文档格式，也是解析难度最大的格式之一。理解 PDF 的解析问题，是掌握文档处理的关键。

**PDF 的本质。** PDF（Portable Document Format）的设计初衷是"在任何设备上看起来都一样"。它的核心是排版信息，不是内容信息。一个 PDF 文件记录的是"在页面坐标 (x, y) 处用 12pt 的 Times New Roman 字体渲染文字'Hello'"，而不是"这里有一段文本，内容是'Hello'"。

这个设计目标导致了 PDF 解析的根本困难：PDF 不关心内容的逻辑结构，只关心视觉呈现。你要从视觉信息中反推内容结构，这本质上是一个"逆向工程"。

**PDF 解析的三种方式：**

第一种是文本提取。直接从 PDF 文件中提取嵌入的文本信息。这适用于"文字型 PDF"——就是用 Word 或 LaTeX 生成的、文字可以被选中和复制的 PDF。这种方式速度快、准确率高，但对"扫描型 PDF"（由扫描仪生成的图片 PDF）完全无效。

第二种是 OCR（光学字符识别）。对每一页 PDF 进行图像识别，提取其中的文字。这适用于扫描件、照片中的文字、以及一些无法直接提取文本的 PDF。OCR 的准确率受图像质量、字体、语言、排版复杂度影响很大。现代 OCR 工具在干净的单栏文档上表现已经相当不错，但在多栏排版、手写批注、表格密集的页面上仍然容易出错。

第三种是视觉模型解析。这是最近两年兴起的方法，用多模态大模型（如 GPT-4V、Claude 等视觉模型）直接"看"PDF 页面的图像，理解其中的内容结构和语义。这种方法在处理复杂排版（多栏、嵌套表格、图文混排）时效果最好，但成本也最高，处理速度最慢。

**PDF 解析的具体挑战：**

- **多栏排版。** 学术论文和杂志通常采用双栏排版。文本提取时如果不考虑排版，左栏的最后一行会和右栏的第一行混在一起，导致文本顺序完全错乱。
- **页眉页脚。** 每页重复出现的页眉（文档标题、章节名）和页脚（页码、版权声明）会污染正文内容。如果不去掉，这些重复文本会被当作文档内容参与检索，干扰结果。
- **表格。** PDF 中的表格是视觉上的格子，但文本提取出来后，行列关系会丢失，变成一段毫无结构的文本。而表格中的信息往往是非常重要的数据。
- **图片中的文字。** 很多 PDF 包含嵌入的图片，图片中的文字（如流程图中的标注、截图中的内容）无法通过文本提取获得。
- **公式。** 数学公式在 PDF 中是用特殊字体渲染的，文本提取出来通常是一堆乱码。
- **书签和目录。** PDF 文件可能包含书签信息，这其实是很有价值的结构信息，但简单的文本提取会忽略它。

**工程建议。** 对于文字型 PDF，优先用文本提取（如 PyPDF2、pdfplumber）。对于扫描型 PDF，必须用 OCR（如 Tesseract、PaddleOCR）。对于复杂排版（学术论文、报表），考虑用视觉模型解析（但成本高）。页眉页脚要在清洗阶段去除，表格需要专门的解析器（如 Tabula）。

### 二、Markdown 解析

Markdown 是对 RAG 系统最友好的文档格式，没有之一。

**为什么 Markdown 是最友好的？** 因为 Markdown 本身就是纯文本，天然带有结构标记。# 表示一级标题，## 表示二级标题，- 表示列表项，| 表示表格行。这些标记既人类可读，又机器可解析。你不需要做任何"逆向工程"，直接读取文件内容就行。

**Markdown 的解析要点：**

- **标题层级。** Markdown 的标题（#、##、### 等）天然提供了文档的结构信息。解析时应该提取标题层级，作为后续切分的边界依据。
- **代码块。** 用三个反引号包围的代码块需要特殊处理。代码块不应该被当作普通文本切分，因为它有自己完整的逻辑含义。
- **表格。** Markdown 表格用 | 分隔，解析相对容易。但要注意表格可能很宽或很长，直接作为一个整体文本块可能太大。
- **链接和图片。** Markdown 中的链接 [text](url) 和图片 ![alt](url) 需要解析。链接的 URL 可能包含有价值的信息，图片的 alt 文本可能包含描述。
- **元数据。** 很多 Markdown 文件开头有 YAML 格式的元数据块（--- 包围的部分），包含标题、作者、日期、标签等信息。这些元数据应该被提取出来，作为文档级别的元数据存储。

**工程建议。** Markdown 解析用 Python-Markdown 或 mistune 库。提取标题层级作为结构信息。代码块保持完整，不要在代码块中间切分。提取 YAML 元数据块。

### 三、Word 解析

Word 文档（.docx）在企业中也很常见，尤其是合同、报告、制度规范这类文档。

**Word 文件的结构。** .docx 文件本质上是一个 ZIP 压缩包，里面包含 XML 文件。Word 文档的内容、样式、结构都记录在这些 XML 中。python-docx 库可以解析这些 XML，提取文本和结构信息。

**Word 解析的要点：**

- **段落和样式。** Word 文档的段落是基本单元。每个段落有样式信息（标题 1、标题 2、正文、引用等）。样式可以作为结构提取的依据——样式为"标题 1"的段落是一级标题。
- **表格。** Word 表格和 PDF 表格不同，它的行列结构是明确的。解析 Word 表格时应该保留行列关系，或者把表格转换为 Markdown 表格格式。
- **嵌入对象。** Word 文档可能嵌入图片、公式、其他 Office 文件。这些嵌入对象中的文本也需要提取（如果可能的话）。
- **页眉页脚。** 和 PDF 一样，Word 的页眉页脚也需要去除。

**工程建议。** 用 python-docx 库解析。提取段落文本和样式，样式映射到标题层级。表格用专门的逻辑处理，保留结构。去除页眉页脚。

### 四、Excel 解析

Excel 文件（.xlsx）和前面几种格式不同，它不是自然语言文档，而是结构化数据表格。

**Excel 解析的问题。** Excel 的每个单元格可能只包含一个词或一个数字，直接按单元格提取会导致碎片化。RAG 系统需要的是有语义完整性的文本块，不是孤立的数据。

**Excel 解析的策略：**

- **按行文本化。** 把每一行转换成一个自然语言句子。比如一行数据"产品A | 100元 | 库存充足"转换为"产品 A 的价格是 100 元，当前库存充足"。
- **按表格摘要化。** 对于大型表格，不要逐行文本化，而是生成表格摘要。"这个表格包含 50 个产品的价格和库存信息，其中产品 A 价格最高，产品 B 库存最低"。
- **Sheet 级别的元数据。** Excel 文件的 Sheet 名称、表格的标题行（第一行通常是列名）都是有价值的结构信息。

**工程建议。** 用 openpyxl 或 pandas 库解析。判断表格大小，小表格逐行文本化，大表格生成摘要。提取 Sheet 名称和列名作为元数据。

### 五、HTML 解析

HTML 是网页的格式。如果知识库包含网页内容（比如帮助文档、技术博客），就需要解析 HTML。

**HTML 解析的优势。** HTML 有明确的语义标签（<h1>、<h2>、<p>、<ul>、<table>），结构提取比 PDF 容易得多。

**HTML 解析的挑战：**

- **JavaScript 渲染的内容。** 很多现代网页的内容是通过 JavaScript 动态加载的，直接解析 HTML 拿不到这些内容。需要用浏览器渲染工具（如 Playwright、Selenium）获取最终渲染后的 HTML。
- **导航和广告。** 网页通常包含导航栏、侧边栏、页脚、广告等内容，这些对知识库价值不大，需要去除。
- **重复内容。** 同一个网站的多个页面可能包含相同的页脚、版权声明，这些重复内容会污染检索结果。

**工程建议。** 用 BeautifulSoup 或 lxml 库解析静态 HTML。对于 JS 渲染的内容，用 Playwright 获取。去除导航栏、侧边栏、页脚等非正文内容。

### 六、文本清洗

解析出文本后，下一步是清洗。清洗的目标是去除噪声，保留有价值的内容。

**需要去除的噪声：**

- **页眉页脚。** 每页重复出现的文档标题、章节名、页码、版权声明。
- **多余空白。** 连续的空格、换行符、制表符。合并为单个空格或单个换行。
- **特殊字符。** 不在常用字符集内的符号、乱码、控制字符。
- **页码和分节符。** 孤立的数字、"第 X 页"、"---" 等分页符号。
- **版权和免责声明。** 长篇的版权声明、免责条款，通常对问答没有帮助。

**清洗的原则：**

- **宁可少去，不要误杀。** 有些内容看起来像噪声，但实际上可能包含关键信息。清洗时要保守。
- **保留结构标记。** 标题、段落、列表的分隔符要保留，它们是切分和检索的基础。
- **可逆操作。** 如果可能，保留原始文本和清洗后文本的映射关系。当发现清洗过度时，可以回退。

**工程建议。** 用正则表达式去除页眉页脚（识别重复模式）。用正则表达式合并多余空白。特殊字符可以定义一个白名单，只保留白名单内的字符。建立清洗前后的对比日志，方便调试。

### 七、结构信息提取

清洗后的文本是平铺的字符串，但原始文档有结构。提取结构信息对于后续切分和检索非常重要。

**需要提取的结构信息：**

- **标题层级。** 文档有几级标题，每个标题的级别是什么，标题之间的父子关系。
- **段落边界。** 哪些文本属于同一段落。
- **列表结构。** 哪些文本是列表项，列表的嵌套层级。
- **表格位置。** 表格在文档中的位置，表格的标题和说明。

**元数据提取：**

- **文档级别元数据。** 文档标题、作者、创建日期、修改日期、文档类型、所属分类。
- **片段级别元数据。** 每个文本片段所属的章节、页码、在原文中的位置。

**工程建议。** 对于 Markdown 和 Word，从样式和标记中提取结构。对于 PDF，从字体大小和位置推断标题（字体大的是标题）。提取的元数据以 JSON 格式存储，和文本一起入库。

## 今日总结

文档解析是 RAG 管线的第一步，也是最容易被低估的环节。

PDF 是最难解析的格式，需要区分文字型和扫描型，注意多栏排版、表格、页眉页脚。

Markdown 是最友好的格式，天然有结构标记，解析简单。

Word 需要从样式映射到结构，表格需要保留行列关系。

Excel 不是自然语言，需要文本化或摘要化处理。

HTML 需要处理 JS 渲染内容，去除导航和广告。

文本清洗要去除噪声但保留有价值内容，原则是宁可少去不要误杀。

结构信息和元数据是后续切分和检索的基础，从第一天就要设计好。
