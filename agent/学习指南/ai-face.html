AI 应用开发面试题库

基于 8 周 56 天学习计划整理，覆盖基础概念、工程实践、架构设计、商业化四大板块。

一、LLM 基础原理

基础

Q1：用一句话概括大语言模型的本质。为什么说它是”概率生成系统”而不是”知识库”？ ⭐

大模型是文字接龙机器，根据上下文概率预测下一个词。它存储的是词语间的统计关系，不是事实本身。Day02

Q2：Token 和”词”有什么区别？为什么计费按 Token 而不是字数？ ⭐

Token 是模型自定义的文本切分方式，介于字符和词之间。英文常见词可能 1 个 Token，不常见词被拆成多个。中文 1 字约 1-2 个 Token。按 Token 计费是因为这是模型实际的计算单位。Day02

Q3：什么是 Context Window？128K 的窗口实际可用空间大约多少？ ⭐

模型单次能处理的最大 Token 数。实际可用空间 = 总窗口 - System Prompt - 对话历史 - 输出预留。且模型对中间位置内容有”遗忘效应”。Day02

Q4：预训练、指令微调、RLHF 三个阶段分别解决什么问题？ ⭐⭐

预训练：在海量文本上学习语言能力（“博览群书”）。指令微调：学会按指令行事（“学会听话”）。RLHF：对齐人类偏好，让回答更安全更有用（“学会说人话”）。Day02

Q5：Transformer 为什么替代了 RNN？Attention 机制解决了什么问题？ ⭐⭐

RNN 必须顺序处理，无法并行且会遗忘长距离内容。Transformer 用 Attention 同时关注所有词，可并行计算，能捕捉长距离依赖。Day02

进阶

Q6：幻觉（Hallucination）有哪几种表现形式？为什么不可能完全消除？ ⭐⭐

事实性幻觉（编造不存在的事实）、来源幻觉（引用不存在的论文）、能力幻觉（声称做了做不到的事）、推理幻觉（逻辑看似合理但结论错误）。根源是概率生成机制，只能缓解不能根除。Day02

Q7：Temperature 从 0 到 2 分别适合什么场景？为什么设为 0 也不能保证 100% 确定性？ ⭐⭐

0-0.3 精确任务（数据提取、分类），0.3-0.7 平衡任务（翻译、总结），0.7-1.2 创意任务（写作、头脑风暴）。设为 0 仍有浮点精度差异导致的微小不同。Day02/Day08

Q8：大模型和数据库、搜索引擎在信息处理上的根本区别是什么？ ⭐⭐

数据库是确定性存储检索，搜索引擎是索引检索返回网页链接，大模型是概率生成直接给答案。数据库精确但死板，搜索引擎可溯源但需人工筛选，大模型直接但不可靠。企业应用中三者常结合使用。Day02

Q9：什么是涌现能力？它对模型选择有什么影响？ ⭐⭐⭐

模型参数量超过阈值后突然展现小模型不具备的能力（复杂推理、多语言、代码生成）。影响：不是所有任务都需要大模型，但复杂推理任务确实需要足够大的模型。Day02

二、Prompt 工程

基础

Q10：Prompt 的作用和局限分别是什么？ ⭐

作用：意图传达、上下文提供、格式约束、质量标准、边界划定。局限：不能改变模型能力边界、不能保证输出稳定、不能获取实时信息、不能执行动作、不能自我验证。Day03

Q11：System Prompt 和 User Prompt 各自放什么内容？ ⭐

System Prompt 定义”你是谁、你怎么工作”（角色、行为规范、输出格式），整个对话不变。User Prompt 定义”你现在要做什么”（具体任务、输入数据、补充要求）。Day03

Q12：什么是 Few-shot Prompting？它和 Zero-shot 的区别？ ⭐

在 Prompt 中提供几个输入输出示例，让模型学习期望模式。Zero-shot 不给示例只给指令。复杂任务 Few-shot 效果通常更好，但会多占 Token 空间。Day03

进阶

Q13：Role-Task-Context-Constraint 框架中，缺少任何一个会导致什么问题？ ⭐⭐

缺 Role：输出风格和视角不明确。缺 Task：模型不知道具体要做什么。缺 Context：模型缺乏足够信息，输出空洞。缺 Constraint：模型可能跑偏、编造、格式不对。Day03

Q14：什么是 Prompt Contract？它和普通 Prompt 写法有什么本质区别？ ⭐⭐

把 Prompt 当作契约来设计：输入约定（提供什么）、输出约定（返回什么）、边界约定（异常怎么处理）。本质区别是从”随便写写”变成”可管理、可测试、可迭代的规范”。Day03

Q15：企业级 Prompt 应该包含哪些部分？为什么需要版本管理？ ⭐⭐

元数据、输入定义、Prompt 模板、Few-shot 示例、质量标准、边界处理、测试用例、变更记录。版本管理是为了支持回滚、评估效果、团队协作和问题复现。Day03

Q16：Prompt 里写”请输出 JSON”和用 JSON Schema 约束有什么区别？ ⭐⭐

前者是软约束，模型大多数时候照做但不保证。后者是硬约束，通过机制保证输出格式。JSON Schema 还能校验字段类型、必填性、值范围。企业应用必须用后者。Day03/Day04

三、结构化输出

基础

Q17：为什么企业应用不能只靠自然语言输出？ ⭐

自然语言解析困难、格式不稳定、信息可能遗漏、边界模糊、扩展性差。程序需要精确、格式化、可预测的输入。Day04

Q18：JSON Mode 和 JSON Schema 有什么区别？ ⭐

JSON Mode 只保证输出合法 JSON，不控制结构。JSON Schema 进一步约束字段名、类型、必填性、值范围、枚举值。Day04/Day09

Q19：Pydantic 在结构化输出流程中扮演什么角色？ ⭐

用 Python 类型注解定义数据结构，自动生成 JSON Schema，自动校验模型输出，提供友好的错误信息。是 JSON Schema 和业务代码之间的桥梁。Day04

进阶

Q20：结构化输出失败有哪些场景？各自的策略是什么？ ⭐⭐

JSON 不合法 → 格式修复或反馈模型修正。结构不符 Schema → 重试并附带错误信息。内容不达标 → 另一个模型评估后重试。输出截断 → 增大 Max Tokens 或简化输出结构。Day04

Q21：什么是”自我修正”重试？它比盲目重试好在哪？ ⭐⭐

把上一次的错误信息反馈给模型，让它知道哪里做错了再修正。比盲目重试更高效，因为模型能针对性地修复问题。Day04

Q22：设计 JSON Schema 时应该从什么出发？ ⭐⭐

从下游系统的需求出发，而不是模型能输出什么。确保每个字段都是下游需要的，同时保持简洁——嵌套不超过三层，能用扁平结构就不嵌套。Day04

四、模型选择与路由

基础

Q23：通用对话模型和推理模型的核心区别？各适合什么场景？ ⭐

通用模型快速给出答案（“快但可能错”），推理模型先内部思考再回答（“慢但对”）。推理模型适合数学、逻辑推理、复杂规划，调用成本更高。Day05

Q24：Embedding 模型和 Reranker 模型在 RAG 中各自的角色？ ⭐

Embedding 把文本变成向量用于初始检索（快但粗），Reranker 对检索结果重新排序（慢但精）。RAG 通常两者配合：先向量检索 top-20，再 Reranker 排序选 top-5。Day05

Q25：云模型和本地模型各有什么优缺点？ ⭐⭐

云模型：能力强、使用简单，但数据要发到第三方、按量计费。本地模型：数据不出本地、不计调用费，但需要 GPU 硬件、运维成本高、能力通常不如顶级商业模型。Day05

进阶

Q26：什么是模型路由？有哪些常见策略？ ⭐⭐

根据任务特征自动选择合适模型。策略：按任务类型路由、按复杂度路由（简单用小模型）、按成本路由（优先低成本）、按延迟路由（实时用轻量模型）、动态路由（根据负载实时调整）。Day05

Q27：一个 RAG 系统通常需要哪些类型的模型？ ⭐⭐⭐

Embedding 模型（文档向量化）、Reranker 模型（检索结果重排序）、通用对话模型（基于检索结果生成回答），可能还需要多模态模型（处理图片/扫描件）。Day05

五、API 调用与工程实践

基础

Q28：API Key 为什么不能硬编码？应该怎么管理？ ⭐

硬编码提交到 Git 后会被自动扫描滥用。应该存环境变量，不同环境用不同 Key，定期轮换，不同项目用独立 Key。Day08

Q29：Streaming 输出和同步输出有什么区别？什么场景该用 Streaming？ ⭐

同步：等模型全部生成完一次性返回。Streaming：逐 Token 发送，用户实时看到内容。面向用户的场景都应该用 Streaming，后台批处理可以用同步。Day08

Q30：什么类型的 API 错误可以重试？什么不能？ ⭐

可重试：速率限制 429、服务端错误 5xx、网络超时。不可重试：认证错误 401、权限错误 403、请求参数错误 400。Day08

进阶

Q31：指数退避重试为什么要加随机抖动（Jitter）？ ⭐⭐

防止多个客户端同时被限流后在同一时刻重试，造成”惊群效应”再次冲击服务端。加上随机抖动让各客户端的重试时间错开。Day08

Q32：速率限制（Rate Limit）从哪些维度限制？怎么应对？ ⭐⭐

维度：每分钟请求数 RPM、每分钟 Token 数 TPM、并发请求数、每日/月用量上限。应对：请求队列、令牌桶算法、多 Key 轮询、分级处理。Day08

Q33：设计 LLM Client 时为什么要考虑多模型支持？ ⭐⭐

不同任务需要不同模型，模型性价比在不断变化，服务商可能出故障需要降级切换。好的 Client 通过适配器模式隔离服务商差异，业务代码只说”做什么”不关心”用哪个模型”。Day08

Q34：Function Calling 的完整流程是什么？ ⭐⭐

定义工具 Schema（名称、描述、参数）→ 用户提问 → 模型判断是否需要调工具 → 模型输出工具调用请求（工具名+参数）→ 程序执行工具 → 把工具结果返回给模型 → 模型生成最终回答。Day10

六、RAG 系统

基础

Q35：RAG 解决什么问题？它的标准流程是什么？ ⭐

解决大模型不知道企业内部信息的问题。流程：用户提问 → 问题向量化 → 在文档库中检索相关片段 → 把片段和问题一起传给模型 → 模型基于真实文档生成回答。Day15

Q36：为什么要对文档做切块（Chunking）？切分策略有哪些？ ⭐

文档太长超过 Context Window 必须切分，且小段落检索精度更高。策略：固定长度切分、按段落切分、按结构切分（标题/章节）、语义切分。需要设置 Overlap 防止信息被切断。Day17

Q37：向量数据库选型要考虑哪些因素？ ⭐

向量维度支持、相似度计算方式（余弦/欧氏/点积）、检索速度、是否支持元数据过滤、是否支持混合检索（向量+关键词）、部署方式（云/本地）。Day18

进阶

Q38：RAG 的常见优化手段有哪些？按优先级排序。 ⭐⭐

  1. Query Rewrite（查询改写，让检索更准确）2) Hybrid Search（混合向量+关键词检索）3) Rerank（重排序检索结果）4) Context Compression（压缩上下文）5) Multi-query（多角度检索）6) Answer Grounding（答案定位到原文）7) Citation Check（引用验证）。Day20

Q39：RAG 系统检索失败有哪些回退策略？ ⭐⭐

检索结果为空：告知用户没有相关文档，降低回答。检索结果不相关：扩大检索范围或换关键词重试。检索结果太多：用 Reranker 精选 top-k。所有策略都失败：回退到模型自身知识回答并标注”未找到相关文档”。Day20

Q40：如何评估 RAG 系统的检索质量？ ⭐⭐⭐

指标：召回率（相关文档是否被检索到）、准确率（检索到的文档是否真的相关）、回答一致性（回答是否基于检索到的文档）、引用准确率（标注的引用是否正确对应）。方法：构建标注数据集、用自动化评估+人工抽检结合。Day19/Day20

Q41：RAG v1 的典型局限是什么？ ⭐⭐

只做单次检索、不支持多轮追问、不支持混合检索、没有查询改写、长文档处理粗糙、引用可能不准确。这些都是后续优化的方向。Day19

七、Agent 与 Workflow

基础

Q42：Workflow 和 Agent 的核心区别？各适合什么场景？ ⭐

Workflow 流程预先定义、固定不变，适合标准化任务（报告生成、数据处理）。Agent 实时规划、灵活决策，适合非标任务（复杂研究、异常处理）。Workflow 遇到未定义情况会卡住，Agent 会尝试自己解决。Day06/Day22/Day23

Q43：Agent 的四个核心组件是什么？执行循环是怎样的？ ⭐

目标（要完成什么）、工具（能调什么）、记忆（知道什么、做过什么）、规划（决定下一步）。循环：理解状态 → 制定计划 → 执行动作 → 观察结果 → 调整计划 → 重复直到完成。Day23

Q44：Multi-Agent 和单 Agent 相比有什么优势？不是越多越好为什么？ ⭐⭐

优势：每个 Agent 专注擅长领域，分工协作处理复杂任务。不是越多越好因为：通信开销增大、结果整合困难、系统复杂度飙升、调试更难。要找到合适的分工粒度。Day25

进阶

Q45：Human-in-the-loop 有哪几种模式？什么场景必须加人工审核？ ⭐⭐

模式：审批式（Agent 等人工确认后执行）、监督式（人工可以随时介入）、抽检式（只检查部分结果）。高风险操作（扣款、删除、对外发送）、低置信度输出、首次上线的任务必须加人工审核。Day26

Q46：Agent 有哪些常见的失败模式？怎么预防？ ⭐⭐

死循环（反复做同一件事）、目标漂移（偏离原始目标）、资源耗尽（调用太多工具/Token）、权限越界（调用了不该调的工具）。预防：设最大步数限制、定期检查与原始目标的对齐程度、设 Token 预算、工具调用权限白名单。Day27

Q47：Agent 安全面临哪些威胁？ ⭐⭐⭐

Prompt 注入（用户精心构造输入绕过限制）、工具越权（Agent 调用不该调的工具）、数据泄露（Agent 输出了敏感信息）、间接注入（知识库中被植入恶意指令）。防护：输入过滤、工具权限白名单、输出审查、知识库安全审计。Day27

八、AI 应用工程化

基础

Q48：AI 应用系统的九大模块分别是什么？ ⭐

输入层、模型层、知识层、工具层、流程层、记忆层、评估层、安全层、交付层。不是每个应用都需要全部，但要知道每个解决什么问题。Day01

Q49：为什么说 Prompt 不是 AI 应用的全部？ ⭐

Prompt 不能保证输出稳定、不能获取实时信息、不能执行动作、不能保证安全、不能规模化。AI 应用是系统工程，Prompt 只是其中一个组件。Day01/Day03

Q50：一个后端 API 从接收请求到返回结果的完整链路是什么？ ⭐⭐

接收请求 → 参数校验 → 构建 Prompt → 调用模型 → 解析结构化输出 → 字段校验 → 结果入库 → 返回响应。每一步都有错误处理和日志记录。Day12/Day13

进阶

Q51：Eval 评估体系包含哪些维度？怎么做自动化评估？ ⭐⭐

准确性（内容是否正确）、完整性（是否遗漏）、格式合规（输出格式是否正确）、安全性（是否有不当内容）、一致性（多次调用是否稳定）。自动化评估用另一个模型做评判（LLM-as-Judge），结合人工抽检。Day31

Q52：成本优化有哪些具体手段？ ⭐⭐

模型路由（简单任务用小模型）、缓存相似问题的结果、批量处理减少调用次数、控制输出长度、Prompt 精简减少输入 Token、选择合适的 Embedding 模型维度。Day32

Q53：AI 应用部署要注意什么？ ⭐⭐

环境隔离（开发/测试/生产）、配置管理（API Key、模型参数）、监控告警（调用成功率、延迟、Token 消耗、成本）、日志收集、灰度发布、回滚机制。Day34

九、行业方案与商业化

基础

Q54：行业拆解方法论的核心步骤是什么？ ⭐

选定行业 → 画产业链（上游/中游/下游）→ 找关键环节 → 分析每个环节的痛点 → 评估 AI 机会。关键是从具体环节切入，不要泛泛谈”AI 提效”。Day36

Q55：岗位拆解从哪几个维度分析？ ⭐

岗位目标、日常任务列表（频率、耗时）、核心痛点、AI 机会（场景描述、方案、预期效果、实施难度）。分析线索：岗位目标 → 日常任务 → 高频痛点 → AI 机会。Day37

进阶

Q56：AI 机会评分模型包含哪些维度？ ⭐⭐

技术可行性（现有技术能不能做）、业务价值（解决了多大的痛点）、实施难度（需要多少资源）、数据就绪度（有没有足够的数据）、ROI 预期（投入产出比）。Day39

Q57：产品包设计包含哪些要素？怎么定价？ ⭐⭐

服务范围、交付物清单、技术方案概述、项目周期、报价。定价方式：按项目（定制化）、按订阅（SaaS）、按调用量（API）。关键是算清成本（模型调用+人力+基础设施）再加利润。Day44

Q58：客户访谈的目的是什么？怎么做需求诊断？ ⭐⭐

目的：理解真实业务场景、确认痛点优先级、评估客户预算和决策权、建立信任。诊断：从业务现状出发（不是从技术出发），问具体场景和数字，识别”想要 AI”和”需要 AI”的区别。Day46

Q59：交付 SOP 的核心步骤是什么？ ⭐⭐

需求确认 → 方案设计 → 数据准备 → 系统开发 → 测试验证 → 用户培训 → 上线运行 → 持续优化。关键是每一步都有明确的交付物和验收标准。Day47

十、综合实战

进阶

Q60：从客户说”我想要一个 AI 助手”到交付方案，你完整的思考过程是什么？ ⭐⭐⭐

  1. 需求澄清：什么业务场景？用户是谁？期望什么效果？
  2. 架构选择：判断是 Chatbot/RAG/Workflow/Agent 哪个级别
  3. 技术方案：需要哪些模块、什么模型、数据怎么准备
  4. 成本估算：模型调用费+开发人力+基础设施
  5. 风险评估：数据安全、模型幻觉、用户接受度
  6. 报价和交付计划 Day50

Q61：一个完整的 AI 项目从 PRD 到上线需要经过哪些阶段？ ⭐⭐⭐

PRD（需求文档）→ 技术方案设计 → 后端开发（API、Prompt、结构化输出）→ 前端开发（交互、报告页）→ Agent 集成（工具、流程）→ 测试优化（准确率、性能）→ 部署上线 → 持续监控。Day50-56

Q62：怎么设计作品集来展示 AI 能力？ ⭐⭐

选 2-3 个代表性项目，每个项目包含：业务背景（解决了什么问题）、技术方案（用了什么架构和模型）、量化成果（效率提升多少、成本节省多少）、可演示的 Demo。作品集本身也是 AI 能力的展示。Day55

Q63：你怎么包装自己成为”AI 专家”？ ⭐⭐⭐

建立个人定位（哪个行业+哪种 AI 应用）、持续输出内容（文章/案例/教程）、积累可展示的案例、建立方法论（不是只会调 API 而是有系统化的解决问题的方法）、建立行业口碑。Day56

十一、系统设计题

Q64：设计一个企业级 RAG 知识库，你会怎么规划技术架构？ ⭐⭐⭐

输入层：文档上传（支持 PDF/Word/Excel）→ 知识层：解析+切块+向量化+存储 → 检索层：混合检索（向量+关键词）+ Reranker → 模型层：基于检索结果生成回答 → 评估层：自动检查回答质量 → 安全层：输入过滤+输出审查 → 交付层：Web/API/嵌入。还需要：Query Rewrite、多轮对话记忆、引用标注、管理后台。

Q65：怎么让一个 AI Agent 系统既灵活又可控？ ⭐⭐⭐

灵活性：给 Agent 清晰的目标、丰富的工具集、自主决策空间。可控性：最大步数限制、工具权限白名单、Token 预算、关键操作需人工确认、输出审查机制、行为日志全记录。核心是在自由度和约束之间找平衡。

Q66：一个 AI 应用的月度模型调用预算是 5000 元，你怎么分配？ ⭐⭐⭐

先统计各功能的调用量和每次的 Token 消耗。按任务复杂度分级：简单分类用小模型（成本占比 20%）、核心生成用大模型（成本占比 60%）、Embedding/Rerank 等辅助（成本占比 20%）。设置每日预算上限和告警，超出时降级到小模型或缓存。
