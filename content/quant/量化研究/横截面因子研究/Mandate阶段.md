## 1. 阶段定义与核心目的

### 1.1 阶段定义

**Mandate（研究授权）**是横截面因子研究流程的第一个阶段。它的核心目的是在接触任何数据之前，**冻结研究的完整边界**——回答"我们在研究什么、不研究什么、怎么算成功"。

在横截面因子研究中，Mandate 尤其关键，因为横截面任务天然具有"范围模糊"的特征：你面对的不是单一的标的或策略，而是大量标的之间的相对关系，极容易在研究中悄无声息地改变目标。

	### 1.2 横截面 Mandate 的特殊性

与传统单标的时间序列策略不同，横截面因子研究的 Mandate 需要额外冻结：

| 维度 | 时间序列策略 | 横截面因子研究 |
|------|-------------|---------------|
| 研究对象 | 单一标的或固定组合 | N 个标的之间的截面排序 |
| 核心问题 | "买不买" / "什么时候买" | "谁比谁强" / "强多少" |
| 范围风险 | 方向漂移 | Universe 膨胀 + Label 窗口漂移 |
| 成功定义 | 收益是否显著 | 排序能力（IC）是否稳定 |

## 2. 为什么横截面研究需要 Mandate

### 2.1 问题场景

**场景 1：Universe 膨胀**

```
初始定义：Top 50 市值币
研究中途：加入一些小币后分组收益更明显
结果：无法判断收益来自因子还是标的选择偏差
```

**场景 2：Label 窗口漂移**

```
初始定义：5 日前向收益
中途发现：20 日收益看起来更稳定
结果：报告说"5 日因子有效"，实际上用的是 20 日结果
```

**场景 3：任务混淆**

```
初始定义：横截面排序任务（谁涨得多）
中途变成：择时任务（什么时候入场）
结果：一个研究同时回答两个问题，哪个都没答清楚
```

### 2.2 Mandate 的核心价值

- **冻结范围**：确保后续所有阶段在同一个框架内工作
- **防止过拟合**：如果 Mandate 定义得好，过拟合的空间会被大幅压缩
- **可复现性**：任何人拿到同一个 Mandate，应该能独立复现研究
- **决策锚点**：Holdout 阶段的晋级/淘汰决策，必须回到 Mandate 的原始定义

## 3. 定义目标市场

### 3.1 市场选择

必须明确且不可更改地定义研究的目标市场。

**需要冻结的内容**：

```yaml
目标市场定义:
  市场: 加密货币现货市场
  交易所: Binance（以 Binance 价格为基准）
  计价货币: USDT
  交易类型: 现货（不含合约、杠杆）
  时区: UTC
```

### 3.2 流动性分层

横截面研究必须定义流动性门槛，这是 Universe 过滤的基础（在 Data Ready 阶段执行，但在 Mandate 阶段冻结规则）：

| 流动性层级 | 筛选标准 | 典型数量 | 用途 |
|-----------|---------|---------|------|
| L1 核心 | 24h 成交额 > $100M，上市 > 180 天 | 10-20 | 主策略 Universe |
| L2 扩展 | 24h 成交额 > $10M，上市 > 90 天 | 30-60 | 容量扩展测试 |
| L3 观测 | 24h 成交额 > $1M，上市 > 30 天 | 100-200 | 因子 discovery，不参与组合构建 |

### 3.3 市场特征认知

在 Mandate 阶段，必须记录对目标市场的已知特征（影响因子设计）：

- **波动率结构**：加密货币波动率远高于传统资产，因子去极值标准需要更保守
- **周期性**：BTC 减半周期约 4 年， regime 划分需考虑
- **24/7 交易**：无收盘价概念，需要明确定义"日"的起止时间
- **新币上市频率高**：幸存者偏差风险大
- **价格操纵风险**：小市值标的价格可能被操纵

## 4. 定义横截面任务

### 4.1 核心任务类型

横截面因子研究的核心任务必须明确为以下之一：

**任务 A：纯排序任务（Pure Ranking）**
```
问题：在时间 t，哪些标的的未来收益更高？
输出：标的排序（rank）
评估指标：IC、Rank IC、分组收益单调性
```

**任务 B：方向+幅度任务（Direction + Magnitude）**
```
问题：在时间 t，标的 i 的预期收益是多少？
输出：预期收益值（continuous）
评估指标：IC、R²、预测误差
```

**任务 C：分类任务（Classification）**
```
问题：在时间 t，标的 i 属于"涨"还是"跌"？
输出：二分类标签
评估指标：准确率、AUC、Precision/Recall
```

### 4.2 冻结任务定义

```yaml
横截面任务定义:
  任务类型: 任务A（纯排序）
  具体问题: "在每 个 rebalance 日，预测未来 h 日收益的截面排序"
  截面定义: 所有满足流动性门槛的标的
  排序基准: 前向对数收益率
  输出形式: 标的 rank（1 = 预期收益最高）
  评估方法: Spearman Rank IC + 分组收益检验
```

### 4.3 任务边界

**必须明确不属于本研究的范围**：

- ❌ 不做择时（什么时候入场）
- ❌ 不做单标的深度分析
- ❌ 不做衍生品定价
- ❌ 不做做市策略

## 5. 定义 Label Horizon

### 5.1 Label 定义的重要性

Label Horizon（标签窗口）是横截面因子研究中**最容易被悄悄修改**的参数。它直接决定了因子评估的结论。

### 5.2 多周期 Label 设计

一个严谨的横截面研究通常会定义多个 Label Horizon，而不是只看一个：

```python
label_horizons:
  短期: 1日, 3日, 5日     # 检验因子的短期预测能力
  中期: 10日, 20日        # 主评估窗口
  长期: 60日, 120日       # 检验因子的长期信息含量
```

### 5.3 Label 计算规则（必须冻结）

```yaml
label 计算规范:
  收益定义: ln(P_{t+h} / P_t)，使用收盘价
  价格基准: UTC 00:00 的收盘价（或 rebalance 时点的 VWAP）
  复权处理: 不适用（加密货币无分红除权）
  异常值处理: 超过 ±3σ 的收益标记为异常，但不删除
  涨跌停处理: 不适用（加密货币无涨跌停）
  停牌处理: 标的在 Label 窗口内停牌 → 该样本标记为 missing
```

### 5.4 Rebalance 频率

必须与 Label Horizon 协调定义：

| Rebalance 频率 | 典型 Label Horizon | 适用场景 |
|---------------|-------------------|---------|
| 每日 | 1-5 日 | 高频因子，高换手 |
| 每周（周一 UTC 00:00） | 5-20 日 | 中频因子，主流选择 |
| 每月（1 日 UTC 00:00） | 20-60 日 | 低频因子，低换手 |

## 6. 定义成功标准

### 6.1 核心评估指标

横截面因子研究的成功标准必须**在 Mandate 阶段冻结**，后续阶段不能修改。

** Formal Gate 指标（必须全部满足）**：

| 指标 | 最低要求 | 说明 |
|------|---------|------|
| 平均 Rank IC | > 0.03 | 截面排序能力的基本门槛 |
| IC 胜率 | > 55% | 正 IC 月数占比 |
| IC t-stat | > 2.0 | IC 显著性检验 |
| 多空收益（Top vs Bottom） | > 0（年化） | 最朴素的风险收益检验 |
| 最大回撤 | < 30% | 风险控制底线 |

**Audit Gate 指标（参考但不阻断）**：

| 指标 | 期望值 | 说明 |
|------|-------|------|
| IC_IR（信息比率） | > 0.5 | IC 的稳定性 |
| 分组收益单调性 | Top > Top-1 > ... > Bottom | 排序质量的直观检验 |
| 换手率 | < 100% 月均 | 实际可交易性 |
| 因子衰减 | 半衰期 > 5 日 | 因子信息持续期 |

### 6.2 阶段性成功标准

| 阶段 | 成功标准 | 决策 |
|------|---------|------|
| Signal Ready | 单因子 IC > 0.02 且 t-stat > 1.5 | 晋级进入 Train |
| Train | 组合 IC > 0.03 且 t-stat > 2.0 | 晋级进入 Test |
| Test | OOS IC > 0.025 且分组收益单调 | 晋级进入 Backtest |
| Backtest | 扣费后夏普 > 1.0 且最大回撤 < 25% | 晋级进入 Holdout |
| Holdout | 所有指标在独立窗口仍然满足 | 最终晋级 |

### 6.3 统计显著性约定

```yaml
统计检验规范:
  显著性水平: 5%（单侧检验）
  IC 检验方法: t-test（IC 均值 / IC 标准差 * sqrt(T)）
  多重检验校正: 不适用（单因子逐一检验）
  功效分析: 确保样本量足够检测 IC > 0.03（通常需要 > 24 个月数据）
```

## 7. Formal Gate 要求

Mandate 阶段的 Formal Gate 是**后续所有阶段的前提**。必须满足以下所有条件才能进入 Data Ready：

| # | 检查项 | 通过标准 | 验证方式 |
|---|-------|---------|---------|
| 1 | 目标市场已定义 | 市场、交易所、计价货币、时区全部明确 | 文档审查 |
| 2 | 横截面任务已定义 | 任务类型、具体问题、输出形式明确 | 文档审查 |
| 3 | Label Horizon 已冻结 | 至少一个主评估窗口，明确计算规则 | 文档审查 |
| 4 | 成功标准已定义 | Formal Gate 指标阈值明确 | 文档审查 |
| 5 | 数据可得性已确认 | 核心数据源确认可用 | 数据源测试 |
| 6 | 范围边界已明确 | "不属于本研究" 的内容有清晰列表 | 文档审查 |

---

## 8. Audit Gate 检查项

| # | 检查项 | 说明 |
|---|-------|------|
| 1 | 市场微观结构认知 | 是否记录了目标市场的特殊结构（24/7、无收盘价等） |
| 2 | 流动性分层合理 | L1/L2/L3 门槛是否经过初步数据验证 |
| 3 | Label 窗口与再平衡频率匹配 | 窗口过短会导致过度交易，过长会导致信号衰减 |
| 4 | 成功标准不过高/过低 | 与同类型研究的 benchmark 对比 |
| 5 | PM/负责人签字 | Mandate 必须有明确的审批记录 |

## 9. 常见错误与反模式

### 反模式 1：Mandate 过于模糊

```
错误示例：研究"加密货币的 alpha"
问题：alpha 是什么？什么标的？什么时间窗口？全都没说
```

**修正**：Mandate 必须具体到"在 Binance 现货市场，使用周频再平衡，基于 5 日前向收益的横截面排序任务"

### 反模式 2：成功标准过拟合数据

```
错误示例：看了数据后，把 IC 阈值设为 0.028（刚好通过）
问题：阈值应该基于理论预期和行业 benchmark，不是数据
```

**修正**：IC > 0.03 是行业常用的最低门槛，应该在见数据之前就设定

### 反模式 3：Label 窗口漂移

```
错误示例：初始设定 5 日，中途改成 20 日但没更新 Mandate
问题：所有后续阶段的评估基准不一致
```

**修正**：任何参数变更必须走 Mandate 变更流程，并记录变更原因

### 反模式 4：忽视幸存者偏差

```
错误示例：Universe 只看现在还在交易的币
问题：退市币的收益被忽略，高估策略表现
```

**修正**：在 Mandate 中明确说明如何处理退市币（通常使用 point-in-time Universe）

## 10. 实际案例：加密货币横截面动量

### Mandate 文档示例

```yaml
研究名称: 加密货币横截面动量因子研究
日期: 2026-04-01
负责人: Researcher A
审批人: PM B

目标市场:
  市场: 加密货币现货
  交易所: Binance
  计价货币: USDT
  时区: UTC
  数据源: Binance REST API（Kline + Ticker）

横截面任务:
  类型: 纯排序任务
  问题: "在每周一 UTC 00:00，哪些标的的未来 7 日收益更高？"
  截面定义: L1 核心 Universe（24h 成交额 > $50M，上市 > 90 天）
  排序基准: 7 日前向对数收益率
  输出: 标的 rank

Label Horizon:
  主评估: 7 日（与再平衡频率匹配）
  辅助评估: 1 日, 3 日, 14 日, 28 日

Rebalance:
  频率: 每周一 UTC 00:00
  价格基准: 再平衡时点的最近 1 小时 VWAP

成功标准:
  平均 Rank IC: > 0.04
  IC 胜率: > 55%
  IC t-stat: > 2.0
  年化多空收益: > 15%
  最大回撤: < 25%
  夏普比率: > 1.5（Backtest 阶段）

不研究:
  - 不做 BTC/ETH 单标的方向性预测
  - 不做合约杠杆策略
  - 不做高频（日频以下）交易
  - 不做跨交易所套利

风险认知:
  - 加密货币市场 24/7 交易，"周"的定义需要统一
  - 小市值币流动性差，滑点大
  - 新币上市频繁，幸存者偏差风险
  - 价格操纵在小市值币中更常见
```

## 12. 与 Data Ready 的交接标准

Mandate → Data Ready 交接清单：

```
✅ 目标市场定义已冻结（市场、交易所、计价、时区）
✅ 横截面任务已明确（任务类型、具体问题、输出形式）
✅ Label Horizon 已定义（主评估窗口 + 计算规则）
✅ Rebalance 频率已冻结
✅ 成功标准已量化（IC、收益、回撤等阈值）
✅ Universe 流动性门槛已设定
✅ 数据源可用性已确认
✅ 研究范围边界已明确
✅ PM 审批通过
```

Data Ready 阶段将基于 Mandate 中定义的 Universe、时间范围和数据需求，构建可研究的数据面板。
