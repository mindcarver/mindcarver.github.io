# Embedding 与向量数据库

## 文章导读

前两天我们把文档解析干净、切成了合适的 Chunk。这些 Chunk 现在还是普通的文本字符串。今天要把它们变成可以计算相似度的数学表示——向量。这就是 Embedding（嵌入）做的事情。

然后这些向量需要一个"家"来存储和检索，这就是向量数据库。

今天是 RAG 系统中最"数学"的一天，但不要被数学吓到。我们的目标是理解 Embedding 的工程意义、知道怎么选择模型和数据库、能完成从文本到向量到检索的完整闭环。底层的数学原理知道概念就够了，不需要推导公式。

学完今天的内容，你应该能够解释"为什么两段文本语义相似，它们的向量就接近"，能够在几个主流向量数据库中做出选型决策，并且完成 Chunk 向量化和语义检索的完整流程。

## 核心概念

### 一、Embedding 工作原理

Embedding 的核心任务是把文本映射成数值向量。一段文本（一个词、一句话、一个 Chunk）经过 Embedding 模型处理后，输出一个固定长度的浮点数数组，比如 [0.023, -0.157, 0.891, ...]。这个数组就是这段文本的向量表示。

**向量的维度。** 常见的 Embedding 模型输出的向量维度从 384 到 3072 不等。维度越高，理论上能表达越多的语义信息，但计算和存储成本也越高。大多数实用场景中，768 到 1536 维是一个合理的范围。

**语义空间的直觉。** Embedding 把文本映射到一个高维空间中，这个空间中语义相似的文本距离近，语义不同的文本距离远。你可以把它想象成一张地图：北京和天津在地图上离得很近（地理上相似），北京和纽约离得很远（地理上不同）。同理，"退款流程"和"退货步骤"在语义空间中离得近（意思相近），"退款流程"和"天气预报"离得远（意思不相关）。

**这是怎么做到的？** Embedding 模型在海量文本上训练，学习到了词语和句子之间的语义关系。训练过程中，经常一起出现的词（比如"退款"和"退货"）被映射到相近的位置，而不相关的词（比如"退款"和"天气"）被映射到较远的位置。经过大量训练后，整个语义空间形成了一种"结构"，使得语义关系可以用几何距离来衡量。

**关键洞察：Embedding 是一种压缩表示。** 一段 500 字符的文本，包含的信息量远超 768 个数字能精确表达的。Embedding 做的是一种有损压缩——保留了最核心的语义信息，丢弃了细节。这就是为什么两段文本的向量相似并不等于两段文本完全一样，它们只是在核心语义上接近。

### 二、向量生成

在 RAG 系统中，向量生成有两个场景：离线生成和在线生成。

**离线生成。** 这是在文档入库阶段，把所有 Chunk 都转换成向量。通常是一个批处理任务：把所有 Chunk 文本收集起来，批量调用 Embedding API，把返回的向量和原始文本一起存入向量数据库。

离线生成需要考虑的工程问题：

- **批处理效率。** Embedding API 通常支持批量请求（一次发送多条文本），比逐条请求效率高得多。但批量大小有上限（比如一次最多 2048 条），需要分批处理。
- **成本控制。** Embedding API 按输入 Token 数计费。百万级 Chunk 的向量化是一笔不小的费用。需要估算总成本，可能的话在测试阶段先用小数据集验证效果。
- **缓存策略。** 同一个 Chunk 的 Embedding 结果是确定性的（给定同一个模型）。如果 Chunk 没有变化，不需要重新生成向量。可以通过 Chunk 的哈希值来判断是否需要重新生成。
- **错误处理。** API 调用可能失败（网络错误、速率限制、服务不可用）。需要实现重试机制和断点续处理。

**在线生成。** 这是在用户提问时，把用户的问题转换成向量，用于检索。这是一个实时操作，对延迟敏感。

在线生成的工程要求：

- **低延迟。** 用户等不了太久。Embedding API 的调用时间通常在几十到几百毫秒，需要选择响应快的 API 端点。
- **一致性。** 离线和在线必须使用同一个 Embedding 模型。如果文档用模型 A 生成向量，用户问题用模型 B 生成向量，两个模型生成的向量空间不同，相似度计算就没有意义。

### 三、Embedding 模型选择

**闭源模型 vs 开源模型。**

闭源模型（如 OpenAI 的 text-embedding-3、Cohere 的 embed）通常质量更好、更稳定，但需要付费调用 API。开源模型（如 BGE、E5、GTE）可以自部署，成本更低，但需要自己维护基础设施。

**模型选择的关键指标：**

- **维度。** 影响存储和计算成本。维度越高，存储越大、检索越慢，但语义表达能力可能更强。
- **最大输入长度。** 有些模型限制输入 Token 数（比如 512 Token），超过部分会被截断。Chunk 长度要考虑模型的限制。
- **语言支持。** 有些模型针对英文优化，有些是多语言的。中文 RAG 系统要选择支持中文的模型。
- **性能基准。** MTEB（Massive Text Embedding Benchmark）是 Embedding 模型的权威基准，包含检索、分类、聚类等任务。可以参考 MTEB 排行榜选择模型。

**推荐模型：**

- **OpenAI text-embedding-3-large**。质量高，支持多语言，但需要付费。
- **BGE 系列（BAAI/bge-m3）**。开源中文模型，在中文任务上表现优秀。
- **E5 系列（intfloat/multilingual-e5-large）**。多语言模型，支持中英文。
- **GTE 系列（thenlper/gte-large）**。通用模型，在多种任务上平衡。

### 四、向量数据库

向量数据库是专门用来存储和检索向量的数据库。它和传统数据库的区别在于：传统数据库用精确匹配（SQL 的 WHERE 条件），向量数据库用相似度匹配（计算向量距离）。

**为什么需要向量数据库？**

当然可以用传统数据库存储向量，然后用 Python 计算相似度。但当数据量达到十万、百万级时，这种方式的性能无法接受。向量数据库做了专门的优化：向量索引（如 HNSW、IVF）可以快速找到最近的向量，而不需要和所有向量计算距离。

**向量数据库的核心功能：**

- **向量存储。** 存储向量和关联的元数据（原始文本、来源文档、时间戳等）。
- **向量索引。** 建立索引结构，加速最近邻搜索。
- **相似度搜索。** 给定查询向量，返回最相似的 k 个向量及其相似度分数。
- **元数据过滤。** 在检索时可以基于元数据过滤（比如只检索某个文档内的 Chunk）。

### 五、向量数据库选型

**主流向量数据库对比：**

| 数据库 | 部署方式 | 优势 | 劣势 | 适用场景 |
|--------|----------|------|------|----------|
| Chroma | 本地/云 | 简单易用、Python 原生 | 不适合大规模 | 原型开发、小规模应用 |
| Pinecone | 云托管 | 托管服务、性能好 | 成本高、不开放源码 | 需要托管服务的大规模应用 |
| Weaviate | 自部署 | 功能丰富、模块化 | 配置复杂 | 需要定制化的企业应用 |
| Qdrant | 自部署/云 | 性能好、Rust 实现 | 生态相对较小 | 性能敏感的应用 |
| Milvus | 自部署 | 开源、功能完整 | 部署复杂 | 大规模分布式应用 |

**选型考虑因素：**

- **数据规模。** 小规模（万级 Chunk）：Chroma。中规模（十万到百万级）：Qdrant、Weaviate。大规模（千万级以上）：Milvus、Pinecone。
- **部署偏好。** 想要托管服务：Pinecone。愿意自部署：Qdrant、Weaviate、Milvus。
- **成本预算。** 开源免费（但需要自己运维），托管服务付费（省心）。
- **技术栈匹配。** 如果用 Python 为主，Chroma 最简单。如果用 Rust，Qdrant。

### 六、相似度计算

向量数据库的核心是相似度计算。常见的相似度度量有三种：

**余弦相似度（Cosine Similarity）。** 计算两个向量夹角的余弦值，范围 [-1, 1]，1 表示完全相同，0 表示正交（无关），-1 表示完全相反。余弦相似度只关注方向，不关注大小，是文本向量最常用的相似度度量。

**欧氏距离（Euclidean Distance）。** 计算两个向量在空间中的直线距离。距离越小，越相似。欧氏距离同时关注方向和大小，在图像检索中更常用。

**点积（Dot Product）。** 计算两个向量的对应位置相乘再求和。点积等于余弦相似度乘以两个向量长度的乘积。如果向量已经归一化（长度为 1），点积等于余弦相似度。

**选择建议。** 对于文本 Embedding，用余弦相似度。大多数向量数据库默认也用余弦相似度。

### 七、完整流程

从 Chunk 到检索的完整流程：

1. **Chunk 准备。** 文档解析和切分后得到 Chunk 列表。
2. **批量向量化。** 调用 Embedding API，把所有 Chunk 转换为向量。
3. **存储。** 把向量和元数据存入向量数据库，建立索引。
4. **用户提问。** 接收用户问题。
5. **问题向量化。** 调用同一个 Embedding API，把问题转换为向量。
6. **检索。** 在向量数据库中做相似度搜索，返回 top-k 个 Chunk。
7. **返回结果。** 返回 Chunk 的文本、相似度分数、元数据。

## 核心结论

Embedding 把文本转换为向量，语义相似的文本向量距离近。

向量生成有离线（批量处理 Chunk）和在线（处理用户问题）两个场景。

选择 Embedding 模型要考虑维度、语言支持、性能基准。

向量数据库专门用于向量存储和检索，核心是向量索引加速最近邻搜索。

主流向量数据库包括 Chroma（简单）、Pinecone（托管）、Qdrant（性能）、Weaviate（功能）、Milvus（大规模）。

相似度计算用余弦相似度（文本向量最常用）。

完整流程：Chunk → 向量化 → 存储 → 问题向量化 → 检索 → 返回结果。
