## 1. 资源感知：智能不是无限预算

Agent 的资源包括 Token、时间、API 调用次数、工具成本、上下文窗口、并发额度、人类注意力和外部系统负载。

资源感知优化的核心是：不同任务不应该消耗同样预算。

可以按任务风险和复杂度分层：

- 简单低风险：快速模型、少工具、短上下文。
- 中等复杂：标准模型、必要检索、轻量验证。
- 高风险复杂：强模型、完整计划、多轮验证、人工关口。

如果所有请求都走最高规格，成本不可控；如果所有请求都走低成本路径，高价值任务会失败。

## 2. 推理技术：按问题选择推理深度

推理技术不是越复杂越好。常见推理方式包括分步推理、树状探索、反事实比较、生成-批判循环、约束求解、案例类比等。

选择推理方式时可以问：

- 问题是否需要多步依赖？
- 是否存在多个候选方案？
- 是否需要严格约束满足？
- 是否能用外部验证替代模型自评？
- 失败成本是否值得更深推理？

很多工程任务不需要模型“想很久”，需要的是先读真实文件、运行真实测试、查看真实错误。推理应该服务于证据，而不是替代证据。

## 3. 优先级排序：先做最能降低不确定性的事

Agent 经常面对一堆可能动作：查资料、写代码、跑测试、问用户、重构、部署、回滚。优先级排序决定了系统是否有效。

一个实用排序框架：

```text
先做阻塞项
再做高风险验证
再做高价值产出
最后做润色和优化
```

也可以用“不确定性下降”排序：哪个动作最能减少后续错误，就先做哪个。比如改代码前先读测试，部署前先跑构建，回答业务问题前先确认数据口径。

## 4. 探索与发现：开放式任务要有边界

探索模式适用于目标不完全清晰、路径未知、需要发现新信息的任务，例如市场调研、代码库摸底、论文综述、故障定位、产品机会分析。

探索的风险是发散。一个没有边界的 Agent 可以一直搜索、一直总结、一直发现新问题。

可控探索需要设置：

- 时间预算。
- 来源范围。
- 中间产物格式。
- 停止条件。
- 发现优先级。
- 不确定性标注。

探索不是最终交付，而是为后续决策提供地图。探索结果应该明确哪些是事实、哪些是推断、哪些还需要验证。

## 5. 我的理解

Agent 的“聪明”不是固定属性，而是资源分配策略。

同一个模型，在错误的任务路由、过宽的搜索空间、无验证的自我推理下会显得不可靠；在清晰目标、合适预算、证据优先和分层验证下，会表现得像一个稳定执行者。

资源、推理和探索三件事要一起设计：资源决定能走多远，推理决定怎么走，探索决定往哪里看。
